# Clarify Phase 1 — Static Site Generation (SSG) Design

> **Scope**: MDX-only, static HTML per route + client hydration, Vite 7, zero intermediate artifacts.
> **Out of scope**: OpenAPI, search, i18n, server runtime.

---

## 1. Goals

| # | Requirement | How we satisfy it |
|---|-------------|-------------------|
| 1 | Static deployment: each page has its own HTML file | Build-time SSR writes `dist/<route>/index.html` for every route |
| 2 | Smooth client-side experience after first load | Injected `<script>` hydrates the static markup into a SPA via React Router |
| 3 | MDX rendering only in Phase 1 | Vite plugin discovers `*.mdx` under `docsRoot`, compiles via `@mdx-js/rollup` |
| 4 | Vite 7 (stable), evaluate Rolldown later | Use Vite 7 public API; no rolldown-specific features |
| 5 | Zero intermediate build artifacts | All generation happens in-memory via Vite virtual modules + `ssrLoadModule` |
| 6 | Tailwind via Vite plugin | Keep existing `@tailwindcss/vite` in user's `vite.config.ts` |

---

## 2. Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│  Build Phase (vite build)                                   │
│                                                             │
│  1. Vite bundles client entry (main.tsx) → dist/assets/     │
│                                                             │
│  2. clarifyPlugin buildEnd():                               │
│       a. Discover MDX files under docsRoot                  │
│       b. Build route manifest (virtual:clarify-routes)      │
│       c. Start temp Vite dev server                         │
│       d. ssrLoadModule('@clarify/renderer/server')          │
│       e. For each route: renderToHTML(url) → write HTML     │
│       f. Shutdown temp server                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Runtime (Browser)                                          │
│                                                             │
│  1. Browser receives static HTML (SEO-friendly)             │
│  2. `<script>` loads client bundle                          │
│  3. hydrateRoot mounts React Router over existing markup    │
│  4. Subsequent navigations → client-side SPA transitions    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. Key Design Decisions

### 3.1 Two-Phase Build (Client + SSR) without Disk Artifacts

Holocron uses Spiceflow (RSC framework) which requires a persistent server runtime. Clarify Phase 1 intentionally avoids a server runtime. We generate HTML **at build time only**.

To render React components to HTML strings at build time, we need the server-side `renderToString`. Rather than emitting a separate SSR bundle to disk, we leverage Vite's own SSR module loader:

- **Client bundle**: Normal Vite `build()` outputs JS/CSS assets to `dist/`.
- **SSR execution**: Inside `buildEnd`, create a short-lived Vite server (or reuse the build's `ssrLoadModule` capability) to execute `packages/renderer/source/server.tsx` directly in Node.js. Vite transpiles JSX/TSX on-the-fly in memory.

This gives us:
- ✅ No intermediate `.js` files on disk
- ✅ No separate SSR build step
- ✅ Type-safe, JSX-capable execution environment

### 3.2 Virtual Modules (inspired by Holocron)

Holocron heavily uses virtual modules (`virtual:holocron-config`, `virtual:holocron-navigation`, `virtual:holocron-mdx`). Clarify adopts the same pattern for cleaner separation:

| Virtual Module | Purpose |
|----------------|---------|
| `virtual:clarify-config` | Merged `clarify.json` + plugin options, exposed as runtime config |
| `virtual:clarify-routes` | Route manifest: path → MDX component import map |
| `virtual:clarify-pages/*` | Per-page compiled MDX content (lazy-loaded) |

These modules are resolved by `clarifyPlugin.resolveId()` and generated by `clarifyPlugin.load()`.

### 3.3 Route Discovery

Unlike Holocron (which reads a `docs.json` navigation tree), Clarify Phase 1 uses **file-system based routing** with optional frontmatter overrides:

```
docsRoot/
├── index.mdx           → /
├── getting-started.mdx → /getting-started
├── guides/
│   ├── index.mdx       → /guides
│   └── theming.mdx     → /guides/theming
```

- Routes are derived from file paths relative to `docsRoot`.
- Frontmatter `slug` can override the generated path.
- Frontmatter `hidden: true` removes the page from the nav but keeps the route.

### 3.4 HTML Template

Each generated HTML file follows this structure:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>{pageTitle} | {siteTitle}</title>
    <!-- Vite injects CSS links here during build -->
  </head>
  <body>
    <div id="root">
      <!-- SSR: renderToString output -->
    </div>
    <script type="module" src="/assets/main-{hash}.js"></script>
  </body>
</html>
```

The `<script>` points to the **client bundle** produced by Vite's normal build. CSS is auto-injected by Vite via `import` statements in the client entry.

## 4. Detailed Build Flow

### Step 1: Client Bundle Build

Vite runs the normal build:
- Entry: `apps/docs/source/main.tsx`
- Outputs: `dist/assets/main-{hash}.js`, `dist/assets/index-{hash}.css`, etc.
- The client entry imports `virtual:clarify-routes` and hydrates the app.

### Step 2: Route Manifest Generation

Inside `clarifyPlugin.buildEnd()`:

1. Scan `docsRoot` for `**/*.mdx`.
2. For each file, extract frontmatter (title, description, hidden, slug).
3. Build the route manifest:
   ```typescript
   interface RouteManifest {
     routes: Array<{
       path: string;           // URL path
       slug: string;           // unique identifier
       pageModule: string;     // virtual:clarify-pages/<slug>
       meta: { title: string; description?: string };
     }>;
   }
   ```
4. Register virtual modules so `virtual:clarify-routes` can serve this manifest.

### Step 3: SSR HTML Generation

Still inside `buildEnd()`:

1. Create a temporary Vite server in middleware mode (or use the existing plugin's module graph):
   ```typescript
   const ssrServer = await createServer({
     root: viteConfig.root,
     plugins: [clarifyPlugin], // reuse same plugin instance
     ssr: { noExternal: true },
   });
   ```
2. Load the server renderer:
   ```typescript
   const { renderToHTML } = await ssrServer.ssrLoadModule(
     '@clarify/renderer/server'
   );
   ```
3. For each route in the manifest:
   ```typescript
   for (const route of manifest.routes) {
     const html = renderToHTML({
       config: resolvedConfig,
       routes: manifest.routes,
       url: route.path,
     });
     // Write to disk
     const outFile = path.join(outDir, route.path, 'index.html');
     await fs.mkdir(path.dirname(outFile), { recursive: true });
     await fs.writeFile(outFile, html, 'utf-8');
   }
   ```
4. Close the SSR server.

> **Why a temp server?** `ssrLoadModule` handles JSX/TSX transpilation, alias resolution, and virtual module resolution automatically. No pre-compilation needed.

### Step 4: Asset Linking

The generated HTML must reference the correct client bundle paths. Vite's build manifest (`dist/.vite/manifest.json`) maps logical entry names to hashed file names.

`buildEnd` reads `manifest.json` to determine:
- `main.tsx` → `assets/main-abc123.js`
- `index.css` → `assets/index-def456.css`

These are injected into the HTML `<head>` and `<body>`.

## 5. Client Hydration Flow

The browser receives fully rendered HTML. The client bundle (loaded via `<script>`) executes:

```typescript
// apps/docs/source/main.tsx
import { hydrateRoot } from 'react-dom/client';
import { BrowserRouter } from 'react-router-dom';
import { AppShell } from '@clarify/renderer';
import { config, routes } from 'virtual:clarify-config';

hydrateRoot(
  document.getElementById('root')!,
  <BrowserRouter basename={config.routeBase}>
    <AppShell config={config} routes={routes} />
  </BrowserRouter>
);
```

- `AppShell` renders `<Routes>` with all discovered routes.
- React Router matches the current URL and hydrates the existing server-rendered DOM.
- Subsequent link clicks trigger client-side navigation without full page reloads.

## 7. Vite Plugin Changes

`packages/vite-plugin/source/index.ts` needs the following additions:

### 7.1 New Hooks

```typescript
// Existing hooks: resolveId, load (for virtual modules)

// NEW: configResolved — store resolved paths
configResolved(resolvedConfig) {
  this.distDir = resolvedConfig.build.outDir;
  this.manifestPath = path.join(this.distDir, '.vite', 'manifest.json');
}

// NEW: buildEnd — orchestrate SSG
async buildEnd() {
  if (process.env.SKIP_CLARIFY_SSG) return;

  // 1. Generate route manifest from MDX discovery
  const manifest = await this.generateRouteManifest();

  // 2. Create temporary SSR server
  const ssrServer = await createServer({ /* ... */ });

  // 3. Load server renderer
  const { renderToHTML } = await ssrServer.ssrLoadModule(
    path.resolve(__dirname, '../renderer/source/server.tsx')
  );

  // 4. Read build manifest for asset paths
  const buildManifest = JSON.parse(await fs.readFile(this.manifestPath, 'utf-8'));

  // 5. Render each route
  for (const route of manifest.routes) {
    const appHtml = renderToHTML({ config: this.resolvedConfig, routes: manifest.routes, url: route.path });
    const html = this.wrapHtml(appHtml, { buildManifest, route });
    const outPath = path.join(this.distDir, route.path, 'index.html');
    await fs.mkdir(path.dirname(outPath), { recursive: true });
    await fs.writeFile(outPath, html, 'utf-8');
  }

  // 6. Cleanup
  await ssrServer.close();
}
```

### 7.2 MDX Discovery & Virtual Modules

```typescript
// In load() or a dedicated method:
async generateRouteManifest() {
  const mdxFiles = await glob('**/*.mdx', { cwd: this.docsRoot });
  return {
    routes: mdxFiles.map(file => {
      const slug = fileToSlug(file);
      const frontmatter = parseFrontmatter(fs.readFileSync(path.join(this.docsRoot, file), 'utf-8'));
      return {
        path: frontmatter.slug ?? slugToPath(slug),
        slug,
        pageModule: `virtual:clarify-pages/${slug}`,
        meta: { title: frontmatter.title ?? 'Untitled', description: frontmatter.description },
      };
    }),
  };
}
```

### 7.3 OpenAPI Handling in Phase 1

The `virtual:clarify-openapi` module and `<OpenAPI>` component references exist in the design docs but **will not be implemented** in Phase 1.

- Plugin option `openApi` is accepted but emits a warning: *"OpenAPI support is planned for Phase 2."*
- MDX authors using `<OpenAPI>` will see a fallback component rendering a placeholder message.
- The `design/core.md` §1.2 (OpenAPI Resource Flow) remains valid for Phase 2 planning.

## 8. Comparison with Holocron

| Aspect | Holocron | Clarify Phase 1 |
|--------|----------|-----------------|
| **Framework** | Spiceflow (RSC) | React Router + vanilla React SSR |
| **Runtime** | Requires Node.js/Worker server | Static files only (CDN deployable) |
| **Build output** | `dist/` with server bundle | `dist/` with HTML + JS/CSS assets |
| **MDX compilation** | `@mdx-js/rollup` via custom pipeline | `@mdx-js/rollup` (same) |
| **Virtual modules** | Extensive (`holocron-*`) | Minimal (`clarify-*`) |
| **Config source** | `docs.json` / `holocron.jsonc` | `clarify.json` |
| **Routing** | Config-driven (`docs.json` nav tree) | File-system based + frontmatter overrides |
| **Hydration** | Full RSC hydration | Standard React `hydrateRoot` |
| **Plugins auto-injected** | React, Tailwind, Spiceflow | Tailwind only (React is user's responsibility) |
| **AI exports** | `/llms.txt`, `/docs.zip`, `.md` routes | Phase 2 |

**Lessons taken from Holocron:**
1. Virtual modules are an elegant way to pass build-time data to runtime.
2. Per-page code splitting (separate chunks per MDX page) improves caching.
3. Auto-injecting common plugins reduces user friction, but can conflict with user configs — Clarify keeps React plugin explicit.

## 9. Deployment Model

The output `dist/` is a standard static site:

```
dist/
├── index.html              ← /
├── getting-started/
│   └── index.html          ← /getting-started
├── guides/
│   ├── index.html          ← /guides
│   └── theming/
│       └── index.html      ← /guides/theming
├── assets/
│   ├── main-abc123.js
│   ├── index-def456.css
│   └── ...
└── .vite/
    └── manifest.json
```

Deploy to any static host: Vercel, Netlify, GitHub Pages, Cloudflare Pages, AWS S3.

For `routeBase !== '/'` (e.g., GitHub Pages project sites), the HTML `<base>` tag and React Router `basename` are set accordingly.

## 10. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `ssrLoadModule` in `buildEnd` is slow for many routes | Cache transpiled modules; parallelize route rendering |
| MDX frontmatter parsing adds build time | Use `gray-matter` (fast, battle-tested); cache per git SHA like Holocron if needed |
| Virtual module invalidation in dev HMR | Implement `handleHotUpdate` in plugin (Phase 1.1) |
| Client bundle path resolution fails | Read Vite's `manifest.json` reliably; fallback to known entry patterns |
| Large site → memory pressure in temp SSR server | Close server after rendering; stream HTML writes |

---

## 11. Implementation Order

1. **Virtual modules**: Implement `virtual:clarify-config`, `virtual:clarify-routes`, `virtual:clarify-pages/*` in `vite-plugin`.
2. **Route discovery**: Scan `docsRoot` for `.mdx`, build manifest, expose via virtual module.
3. **Renderer wiring**: Ensure `AppShell` can consume the manifest format.
4. **buildEnd SSR**: Add `buildEnd` hook with temp server + `ssrLoadModule` + HTML generation.
5. **Asset injection**: Read `manifest.json`, inject correct script/link tags into HTML.
6. **Testing**: Build `apps/docs`, verify `dist/` has HTML per route, verify hydration in browser.
7. **Cleanup**: Remove/stub OpenAPI references; update design docs.
