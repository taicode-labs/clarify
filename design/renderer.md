# Clarify Renderer — Rendering Architecture

## Overview

The **Renderer** is the client-side runtime of Clarify. It consumes the processed content (routes, MDX components, API data) generated by `@clarify/vite-plugin` and renders it as an interactive documentation site.

## Rendering Philosophy

- **Client-first SPA**: Runs as a Vite-powered Single Page Application. No SSR hydration complexity.
- **Static Build-Time Data**: The plugin generates a route manifest and pre-compiles MDX pages at build time. The renderer consumes these static artifacts at runtime.
- **Component-Driven UI**: Every MDX element maps to a `@clarify/renderer` primitive. The entire visual layer is composable and themeable.
- **Zero-Runtime-Config**: All configuration is resolved at build time and baked into virtual modules. No runtime config parsing.

## 1. Runtime Architecture

### 1.1 Entry Point

```tsx
// apps/docs/source/main.tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import { BrowserRouter } from 'react-router-dom';
import { ClarifyProvider } from '@clarify/renderer';
import { App } from './App';
import './index.css';

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <BrowserRouter>
      <ClarifyProvider>
        <App />
      </ClarifyProvider>
    </BrowserRouter>
  </StrictMode>
);
```

### 1.2 Application Shell

```tsx
// apps/docs/source/App.tsx
import { Routes, Route } from 'react-router-dom';
import { Layout, MDXProvider } from '@clarify/renderer';
import { routes } from 'virtual:clarify-routes';

export default function App() {
  return (
    <Layout>
      <MDXProvider>
        <Routes>
          {routes.map(route => (
            <Route
              key={route.path}
              path={route.path}
              element={<route.component />}
            />
          ))}
        </Routes>
      </MDXProvider>
    </Layout>
  );
}
```

### 1.3 Layout Component

`Layout` is the top-level shell from `@clarify/renderer`:

```tsx
function Layout({ children }: { children: React.ReactNode }) {
  return (
    <div className="clarify-layout">
      <Header />
      <div className="clarify-body">
        <Sidebar />
        <main className="clarify-main">{children}</main>
        <TableOfContents />
      </div>
    </div>
  );
}
```

**Layout regions**:
| Region | Responsibility | Fixed? |
|--------|---------------|--------|
| `Header` | Logo, search trigger, theme toggle, version switcher | Yes (top) |
| `Sidebar` | Hierarchical navigation tree from `navTree` | Yes (left) |
| `Main` | Rendered page content (MDX or API pages) | Scrollable |
| `ToC` | Heading anchors for current page | Yes (right, desktop only) |

## 2. Virtual Module System

The renderer depends on **virtual modules** generated by the plugin at build time. These are the only bridge between the plugin and the renderer.

### 2.1 `virtual:clarify-routes`

The route manifest and navigation tree.

```typescript
export interface RouteManifest {
  routes: Array<{

export const routes: RouteManifest['routes'];
export const navTree: RouteManifest['navTree'];
```



## 3. Component Mapping (MDXProvider)

MDX components are mapped to `@clarify/renderer` primitives via `MDXProvider` from `@mdx-js/react`:

| MDX Element | Renderer Component | Purpose |
|-------------|-------------------|---------|
| `h1`–`h6` | `Heading` | Styled headings with auto-generated anchor IDs |
| `p` | `Paragraph` | Typography with proper spacing and max-width |
| `pre > code` | `CodeBlock` | Syntax highlighting + copy button + language label |
| `code` (inline) | `InlineCode` | Styled inline code spans |
| `table` | `DataTable` | Responsive, styled tables with horizontal scroll |
| `a` | `SmartLink` | Internal routing (client-side) vs external (new tab) |
| `img` | `Image` | Lazy loading, responsive sizing, caption support |
| `blockquote` | `Callout` | Styled callout boxes (info, warning, danger, tip) |
| `hr` | `Divider` | Thematic breaks |
| `<OpenAPI>` | `OpenAPI` | Embedded API reference (see §7) |
| Custom JSX | User-defined | Registered via plugin `components` option |

### 3.1 CodeBlock Architecture

```tsx
interface CodeBlockProps {
  children: string;
  className?: string; // language-xxx
}

function CodeBlock({ children, className }: CodeBlockProps) {
  const language = className?.replace('language-', '') || 'text';
  return (
    <div className="clarify-codeblock">
      <div className="clarify-codeblock-header">
        <span className="clarify-codeblock-lang">{language}</span>
        <CopyButton text={children} />
      </div>
      <pre className="clarify-codeblock-pre">
        <code>{children}</code>
      </pre>
    </div>
  );
}
```

**Syntax highlighting strategy**: Build-time via Shiki (preferred for zero runtime cost) or runtime via PrismJS (fallback for dynamic content).

### 3.2 SmartLink

Distinguishes internal vs external links:

```tsx
function SmartLink({ href, children }: LinkProps) {
  const isExternal = href?.startsWith('http') || href?.startsWith('//');
  if (isExternal) {
    return <a href={href} target="_blank" rel="noopener noreferrer">{children}</a>;
  }
  return <Link to={href}>{children}</Link>;
}
```

## 4. Navigation System

### 4.1 Sidebar

Generated from `navTree` in the route manifest:

```tsx
function Sidebar() {
  const { navTree } = useClarifyRoutes();
  const currentPath = useLocation().pathname;

  return (
    <nav className="clarify-sidebar" aria-label="Documentation navigation">
      <div className="clarify-sidebar-header">
        <SearchTrigger />
      </div>
      <NavTree nodes={navTree} currentPath={currentPath} />
    </nav>
  );
}
```

### 4.2 NavTree

Recursive component with active state and expand/collapse:

```tsx
interface NavNode {
  title: string;
  path: string;
  children?: NavNode[];
  navOrder?: number;
}

function NavTree({ nodes, currentPath, depth = 0 }: NavTreeProps) {
  return (
    <ul className={depth === 0 ? 'clarify-nav-root' : 'clarify-nav-nested'}>
      {nodes.map(node => (
        <li key={node.path} className={isActive(node.path, currentPath) ? 'active' : ''}>
          <NavLink to={node.path}>{node.title}</NavLink>
          {node.children && <NavTree nodes={node.children} currentPath={currentPath} depth={depth + 1} />}
        </li>
      ))}
    </ul>
  );
}
```

### 4.3 Table of Contents (ToC)

Extracted from heading elements in the current page using an `IntersectionObserver` for scroll spy:

```tsx
function TableOfContents() {
  const headings = usePageHeadings(); // h2, h3 extracted from DOM
  const activeId = useActiveHeading(headings); // intersection observer

  return (
    <nav className="clarify-toc" aria-label="Table of contents">
      <p className="clarify-toc-title">On this page</p>
      {headings.map(h => (
        <a
          key={h.id}
          href={`#${h.id}`}
          className={`clarify-toc-link toc-${h.level} ${activeId === h.id ? 'active' : ''}`}
        >
          {h.text}
        </a>
      ))}
    </nav>
  );
}
```

## 5. State Management

Clarify uses **React Context** for lightweight global state. No external state library needed.

### 5.1 Contexts

| Context | Provided By | Data | Consumers |
|---------|-------------|------|-----------|
| `RouteManifestContext` | `ClarifyProvider` | `routes`, `navTree` | App, Sidebar, Search |
| `ThemeContext` | `ClarifyProvider` | `theme`, `toggleTheme` | All components, Layout |
| `SearchContext` | `ClarifyProvider` | `isOpen`, `query`, `results`, `open`, `close` | SearchTrigger, SearchModal |
| `PageContext` | `PageShell` | `meta`, `headings` | ToC, PageShell, SEO meta |

### 5.2 ClarifyProvider

Single provider composing all contexts:

```tsx
function ClarifyProvider({ children }: { children: React.ReactNode }) {
  return (
    <RouteManifestProvider>
      <ThemeProvider>
        <SearchProvider>
          {children}
        </SearchProvider>
      </ThemeProvider>
    </RouteManifestProvider>
  );
}
```

### 5.3 ThemeContext

```tsx
interface ThemeContextValue {
  theme: 'light' | 'dark';
  toggleTheme: () => void;
}

function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<'light' | 'dark'>(() => {
    const stored = localStorage.getItem('clarify-theme');
    if (stored === 'light' || stored === 'dark') return stored;
    return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  });

  useEffect(() => {
    document.documentElement.setAttribute('data-theme', theme);
    localStorage.setItem('clarify-theme', theme);
  }, [theme]);

  const toggleTheme = useCallback(() => {
    setTheme(t => t === 'light' ? 'dark' : 'light');
  }, []);

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}
```

## 6. Search System

### 6.1 Design Goals

- **Fast**: Instant results, no server round-trip.
- **Accurate**: Full-text search across MDX content, headings, and OpenAPI descriptions.
- **Lightweight**: Search index is a compressed JSON asset generated at build time.

### 6.2 Architecture

```
Build Time: Extract text from all MDX + API docs → Build inverted index (miniSearch) → Emit /search-index.json
Runtime:   Load index on demand → Execute queries in browser → Render results in modal
```

### 6.3 SearchModal

Command palette-style UI triggered by `Cmd+K`:

```tsx
function SearchModal() {
  const { isOpen, close } = useSearch();
  const [query, setQuery] = useState('');
  const index = useSearchIndex(); // lazy-loads /search-index.json
  const results = useMemo(() => index?.search(query) ?? [], [query, index]);

  if (!isOpen) return null;

  return (
    <Modal onClose={close}>
      <input
        autoFocus
        placeholder="Search documentation..."
        value={query}
        onChange={e => setQuery(e.target.value)}
      />
      <SearchResults results={results} onSelect={close} />
    </Modal>
  );
}
```

### 6.4 Search Index Format

```typescript
interface SearchDoc {
  id: string;
  title: string;
  excerpt: string;
  path: string;
  category: 'page' | 'api' | 'heading';
}
```

## 7. OpenAPI Rendering (Embedded)

### 7.1 Philosophy: OpenAPI as a First-Class Citizen

OpenAPI is **not** a separate route system. It is an embedded component (`<OpenAPI>`) available in any MDX page. Authors decide where and how API references appear in their documentation flow.

### 7.2 The `<OpenAPI>` Component

```tsx
import { OpenAPI } from '@clarify/renderer';

// In any MDX page:

// Render a single operation
<OpenAPI operationId="getUser" />

// Render all operations under a tag
<OpenAPI tag="Users" />

// Render the full spec (overview page)
<OpenAPI />
```

#### Props

```typescript
interface OpenAPIProps {
  /** Render a specific operation by operationId */
  operationId?: string;

  /** Render all operations under a tag */
  tag?: string;

  /** When true, renders the full spec overview */
  full?: boolean;

  /** Custom title override */
  title?: string;

  /** Hide the method/path badge */
  hideEndpoint?: boolean;

  /** Collapse request/response sections by default */
  collapsed?: boolean;
}
```

### 7.3 Data Flow

```
vite-plugin (build time)
  │
  ├─ Parse openapi.yaml → OpenAPIResource model
  │
  └─ Emit virtual:clarify-openapi
        │
        ▼
@clarify/renderer (runtime)
  │
  ├─ Import spec from virtual:clarify-openapi
  │
  └─ <OpenAPI operationId="..."> → lookup operation → render
```

### 7.4 OpenAPI Component Set

| Component | Purpose | Used By |
|-----------|---------|---------|
| `OpenAPI` | Main container, handles props dispatch | `<OpenAPI />` |
| `ApiEndpointCard` | Method badge + path + description | `OpenAPI` (single op) |
| `ParametersTable` | Query/path/header params | `ApiEndpointCard` |
| `RequestBodySection` | Schema + example for request body | `ApiEndpointCard` |
| `ResponsesSection` | Status codes + response schemas | `ApiEndpointCard` |
| `SchemaViewer` | Recursive JSON schema rendering | `RequestBodySection`, `ResponsesSection` |
| `CodeExample` | Curl + language SDK examples | `ApiEndpointCard` |
| `TagGroup` | Renders all operations under a tag | `OpenAPI` (tag mode) |
| `SpecOverview` | Full spec summary (tags, versions, servers) | `OpenAPI` (full mode) |

### 7.5 Usage Example in MDX

```mdx
---
title: User API Guide
---

# User API Guide

## Authentication

All user endpoints require a Bearer token.

## Get User by ID

<OpenAPI operationId="getUser" />

## List All Users

<OpenAPI operationId="listUsers" collapsed />

## User Management Overview

For a complete reference, see all user endpoints:

<OpenAPI tag="Users" />
```

### 7.6 Search Integration

OpenAPI operations are indexed alongside MDX content. Search results for API operations link directly to the MDX page + anchor where the `<OpenAPI>` component is embedded.

```typescript
interface SearchDoc {
  id: string;
  title: string;
  excerpt: string;
  path: string;          // e.g., "/user-guide#get-user-by-id"
  category: 'page' | 'api' | 'heading';
}
```

## 8. Performance Optimizations

### 8.1 Code Splitting

Each MDX page is a separate chunk via dynamic imports:

```tsx
const GettingStartedPage = lazy(() => import('virtual:clarify-page/getting-started'));
```

### 8.2 Prefetching

Adjacent routes in the navigation tree are prefetched on hover/focus:

```tsx
function NavLink({ to, children }: NavLinkProps) {
  return (
    <Link
      to={to}
      onMouseEnter={() => prefetchRoute(to)}
      onFocus={() => prefetchRoute(to)}
    >
      {children}
    </Link>
  );
}
```

### 8.3 Asset Optimization

| Asset Type | Strategy |
|------------|----------|
| Images | `<img loading="lazy" />`; small images inlined as data URIs |
| Fonts | `<link rel="preload" />` for critical fonts |
| CSS | Tailwind CSS v4 with `@tailwindcss/vite` for minimal CSS output |
| Search Index | Compressed JSON, loaded on first search trigger only |

### 8.4 React Optimizations

- `React.memo` for pure renderer components (Heading, CodeBlock, etc.).
- `useMemo` for expensive computations (nav tree filtering, ToC extraction).
- `useCallback` for event handlers passed to child components.

## 10. Error Handling

### 10.1 Error Boundaries

```tsx
class PageErrorBoundary extends React.Component<
  { children: React.ReactNode },
  { hasError: boolean }
> {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  render() {
    if (this.state.hasError) {
      return (
        <div className="clarify-error">
          <h2>Failed to render this page</h2>
          <p>Try refreshing the page or navigating back.</p>
        </div>
      );
    }
    return this.props.children;
  }
}
```

### 10.2 404 Page

```tsx
function NotFoundPage() {
  return (
    <DocShell title="Page Not Found">
      <p>The page you are looking for does not exist.</p>
      <Link to="/">Go to homepage</Link>
    </DocShell>
  );
}
```
