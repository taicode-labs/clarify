# 插件机制

Clarify 的插件机制用于在构建期接入团队自己的能力，例如内容转换、导航增强、虚拟模块注入、搜索索引生成、翻译流水线或构建完成后的产物处理。

如果你只是在写文档，不需要理解插件；如果你要把 Clarify 集成到产品级文档平台或内部工程体系，插件是推荐扩展点。

---

## 什么时候使用插件

适合使用插件的场景：

| 场景 | 可以做什么 |
|------|------------|
| 内容治理 | 统一注入免责声明、校验标题结构、补齐元数据 |
| 导航增强 | 按团队规则重排路由、过滤内部页面、补充导航节点 |
| 搜索集成 | 在构建期生成 Pagefind、Algolia 或自研搜索索引 |
| 国际化流程 | 构建前后同步翻译产物，或对缺失翻译做检查 |
| 站点集成 | 输出 sitemap、RSS、埋点配置或内部平台清单 |

不适合用插件处理的事情：

- 单篇文章内容排版：优先使用 MDX。
- 常规站点标题、导航、主题色：优先使用 `clarify.ts` 配置。
- 运行时交互组件：优先作为 React/MDX 组件接入。

## 基本结构

插件是一个带 `name` 和 `hooks` 的对象：

```ts
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const auditPlugin: ClarifyPlugin = {
  name: 'docs-audit',
  hooks: {
    'routes:resolved'({ routes, navigation }) {
      console.log(`Resolved ${routes.length} routes`)
      return { routes, navigation }
    },
  },
}

export default defineConfig({
  title: 'Product Docs',
  plugins: [auditPlugin],
})
```

> 需要插件时推荐使用 `clarify.ts`。`clarify.json` 不能表达函数，因此不适合配置插件。

## 生命周期

Clarify 当前构建流程会调用以下 Hook：

| Hook | 触发时机 | 典型用途 |
|------|----------|----------|
| `content:transform` | Clarify 从 MDX、OpenAPI 规范等内容源提取元数据前 | 展开项目变量、规范化 frontmatter、注入生成内容或校验原始内容 |
| `routes:discover` | 从内容根目录发现路由时 | 添加 OpenAPI 等非 MDX 路由，或替换发现结果 |
| `routes:discovered` | 路由发现后、应用配置路径和导航前 | 丰富已发现路由、添加源文件链接或挂载产物 URL |
| `pages:resolved` | 路由映射为页面后、回写路由前 | 改写页面级 frontmatter 或内容 |
| `routes:resolved` | 路由和导航生成后 | 调整路由、导航节点、tabs 和可见页面 |
| `modules:before` | 虚拟模块生成后、交给 Vite 前 | 注入或修改运行时虚拟模块 |
| `html:transform` | HTML 输出或开发服务返回前 | 注入 HTML 标签、主题启动脚本或客户端入口脚本 |
| `dev:configureServer` | Vite 开发服务器配置时 | 注册本地中间件或仅开发环境使用的接口 |
| `build:assets` | 构建产物写入阶段 | 生成额外静态文件 |
| `build:done` | 构建和 SSG 完成后 | 生成额外产物、上报构建结果 |

插件按照 `plugins` 数组顺序依次执行。前一个插件返回的结果会作为下一个插件的输入。

Clarify 内置的项目变量就是通过 `content:transform` 插件实现的。用户插件也运行在同一个生命周期中，并且会在内置插件之后执行。

## Hook 上下文

每个 Hook 会收到统一的上下文对象：

```ts
type ClarifyHookContext = {
  version: string
  projectRoot: string
  contentRoot: string
  projectConfig: ResolvedProjectConfig
  generateOptions: ResolvedBuildOptions
  routes: ContentRoute[]
  navigation: NavigationTree
  plugins: ClarifyPlugin[]
  get<T>(key: string): T | undefined
  set<T>(key: string, value: T): void
  has(key: string): boolean
  delete(key: string): boolean
}
```

- `projectRoot` 和 `contentRoot`：用于解析项目文件和内容文件的绝对路径。
- `projectConfig`：已经应用默认值后的站点配置，例如 `title`、`routePrefix`、`locales`。
- `generateOptions`：构建运行参数，例如内容目录和输出目录。
- `routes` 和 `navigation`：当前 hook 执行点的路由和导航状态。
- `plugins`：排序后的插件列表。
- `get`、`set`、`has` 和 `delete`：用于在 hook 间协作的小型共享 store。

## 示例：过滤草稿页面

可以在 `routes:resolved` 阶段过滤不希望发布的页面：

```ts
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const hideDraftRoutes: ClarifyPlugin = {
  name: 'hide-draft-routes',
  hooks: {
    'routes:resolved'({ routes, navigation }) {
      return {
        routes: routes.filter(route => !route.path.includes('/drafts/')),
        navigation,
      }
    },
  },
}

export default defineConfig({
  plugins: [hideDraftRoutes],
})
```

如果同时修改 `routes` 和 `navigation`，需要保证两者一致，避免侧边栏指向不存在的页面。

## 示例：转换 MDX 内容

如果需要在 Clarify 提取页面标题、描述、正文标题、章节锚点、接口元数据和搜索文本之前修改内容，可以使用 `content:transform`。输入中包含 `kind`，因此插件可以只处理 MDX 页面，避免误改 OpenAPI 规范：

```ts
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const legalNoticePlugin: ClarifyPlugin = {
  name: 'legal-notice',
  hooks: {
    'content:transform'(page) {
      if (page.kind !== 'mdx') return page

      return {
        ...page,
        content: `${page.content}\n\n<LegalNotice />`,
      }
    },
  },
}

export default defineConfig({
  plugins: [legalNoticePlugin],
})
```

## 示例：构建后生成产物

`build:done` 适合生成额外文件，例如 sitemap、搜索索引或内部平台清单：

```ts
import { writeFileSync } from 'node:fs'
import { join } from 'node:path'
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const buildManifestPlugin: ClarifyPlugin = {
  name: 'build-manifest',
  hooks: {
    'build:done'(ctx) {
      const output = ctx.generateOptions.outputDirectory ?? 'output'
      writeFileSync(
        join(output, 'clarify-build.json'),
        JSON.stringify({ title: ctx.projectConfig.title }, null, 2),
      )
    },
  },
}

export default defineConfig({
  plugins: [buildManifestPlugin],
})
```

## 错误处理

如果插件 Hook 抛出异常，Clarify 会中断当前流程，并给出包含插件名和 Hook 名的错误信息：

```txt
[clarify] plugin "my-plugin" hook "routes:resolved" failed: Error: ...
```

建议插件作者：

- 为插件设置稳定、可读的 `name`。
- Hook 内部只做当前阶段必要的工作。
- 修改路由、导航或虚拟模块时返回完整结果。
- 对外部服务调用设置超时和降级策略。
