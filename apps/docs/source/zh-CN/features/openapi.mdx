# OpenAPI 集成

Clarify 把 OpenAPI 规范当作一等内容源：规范文件可以生成完整 API Reference，也可以在 MDX 指南中嵌入单个接口。

这让 API 文档不再只是“接口列表”，而是可以和教程、认证说明、错误处理、迁移指南一起组成完整的开发者体验。

---

## 你可以用它做什么

| 目标 | 推荐能力 |
|------|----------|
| 发布完整接口列表 | `.openapi.json` / `.openapi.yaml` 自动生成 API Reference |
| 在教程中展示一个接口 | `OpenApiEndpoint` |
| 给完整 Reference 添加前后说明 | `OpenApiPage` |
| 让 AI / LLM 读取原始规范 | 构建输出中的 `.openapi.*` 原始文件和 `llms.txt` |
| 多语言站点维护 API 文档 | 把规范放入语言目录，例如 `source/zh-CN/api.openapi.json` |

---

## 最小示例

把 OpenAPI 文件放入内容目录：

```txt
source/
├── index.mdx
└── api.openapi.json
```

运行构建后，Clarify 会生成：

| 产物 | 说明 |
|------|------|
| `/api` | 完整 API Reference 页面 |
| `/api.openapi.json` | 原始 OpenAPI 规范输出 |
| `llms.txt` 中的 OpenAPI 链接 | 方便 AI 工具发现和读取 |

---

## 工作流

```txt
OpenAPI 规范
    ↓ 构建时解析 / dereference
OpenAPI 注册表
    ↓
API Reference 页面 + MDX 接口嵌入 + 原始规范输出
```

构建时，Clarify 会使用 `@apidevtools/swagger-parser` 解析 OpenAPI 3.0 / 3.1 规范。`$ref`、共享 `components`、JSON/YAML 文件和跨结构引用会先被解析，再交给渲染器。

---

## 推荐信息架构

产品级 API 文档通常不应该只有一个 Reference 页面。更推荐拆成：

1. **API 文档工作流**：讲如何组织规范、指南和接口嵌入
2. **生成 API Reference**：讲规范文件如何变成完整接口页面
3. **在 MDX 中嵌入接口**：讲如何把 operation 放进教程
4. **API 文档示例**：展示真实渲染效果
5. **API Reference**：完整接口列表

本站的 “API 文档” 分组就是按这个结构组织的。

---

## 支持的规范格式

| 格式 | 示例 |
|------|------|
| JSON | `api.openapi.json` |
| YAML | `api.openapi.yaml` |
| YAML 简写 | `api.openapi.yml` |

支持 OpenAPI 版本：

- OpenAPI 3.0.x
- OpenAPI 3.1.x
