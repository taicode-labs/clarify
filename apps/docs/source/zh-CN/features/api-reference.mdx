# 生成 API Reference

API Reference 是 Clarify 的 OpenAPI 专用页面。它适合承载完整接口列表，让读者按接口浏览请求参数、请求体、响应、示例和安全信息。

---

## 什么时候使用 API Reference

当你已经有 OpenAPI 规范时，优先使用自动生成的 API Reference，而不是手写接口文档。

适合场景：

- 对外开放 REST API 文档
- 内部服务接口说明
- SDK / 后端服务的接口基准文档
- 希望接口文档与 OpenAPI 规范保持同步

不适合场景：

- 只想在教程中解释一两个接口
- 需要围绕业务流程写长篇指南
- OpenAPI 规范还没有整理完成

这类场景更适合在 MDX 页面中嵌入单个接口，详见 [在 MDX 中嵌入接口](/features/openapi-embedding)。

---

## 文件命名约定

把 OpenAPI 文件放在 `source` 下，并使用以下扩展名：

- `.openapi.json`
- `.openapi.yaml`
- `.openapi.yml`

例如：

```txt
source/
├── index.mdx
├── guides/
│   └── authentication.mdx
└── api.openapi.json
```

`api.openapi.json` 会生成 `/api` 页面。

---

## 路由生成规则

OpenAPI 文件的路由来自文件路径，`.openapi.*` 后缀会被移除：

| 文件 | 生成页面 |
|------|----------|
| `source/api.openapi.json` | `/api` |
| `source/reference/admin.openapi.yaml` | `/reference/admin` |
| `source/services/billing.openapi.yml` | `/services/billing` |

启用 i18n 时，OpenAPI 文件也遵循语言目录规则：

```txt
source/
└── zh-CN/
    └── api.openapi.json  → /api
```

如果有 `routePrefix`，最终访问路径会加上该前缀，但内容文件本身的路由仍按上表推导。

## 加入侧边栏

如果没有配置 `pages`，Clarify 会按文件树生成导航。

如果你手动维护侧边栏，需要使用 `openapi` 条目引用规范文件：

```json
{
  "pages": [
    {
      "group": "API 文档",
      "pages": [
        {
          "openapi": "api.openapi.json",
          "title": "API Reference",
          "icon": "BookOpenCheck"
        }
      ]
    }
  ]
}
```

`openapi` 的值是相对内容根目录的文件引用，可以保留扩展名。对于多语言站点，写语言无关的路径，例如 `api.openapi.json`，不要写 `zh-CN/api.openapi.json`。

## 构建时解析

构建过程中，Clarify 会：

1. 扫描内容目录中的 OpenAPI 文件
2. 使用 `@apidevtools/swagger-parser` 解析并 dereference 规范
3. 为每个接口生成页面章节，用于右侧目录和搜索入口
4. 将解析后的规范写入运行时 OpenAPI 注册表
5. 为每个 OpenAPI 路由生成静态 HTML
6. 输出原始规范文件，供 `llms.txt` 和复制操作使用

如果规范解析失败，生产构建会失败。这能防止过期或无效的接口文档被发布。

---

## 原始规范输出

OpenAPI 路由会保留原始内容入口。例如：

| 页面 | 原始规范 |
|------|----------|
| `/api` | `/api.openapi.json` |
| `/services/billing` | `/services/billing.openapi.yml` |

当站点部署在子路径下时，`llms.txt` 中的链接会自动带上 `routePrefix`。

---

## 内容质量建议

API Reference 的质量主要取决于 OpenAPI 规范本身。建议至少补齐：

- `info.title`、`info.version`、`info.description`
- `servers`
- `tags`
- 每个 operation 的 `summary` 和 `description`
- 参数的 `description`、`schema`、`example` 或 `examples`
- 请求体和响应的 media type
- 主要响应码和错误响应
- 复用的 `components.schemas`
- 认证方式 `components.securitySchemes`

Clarify 会尽量渲染规范中已有的信息，但不会替你补全缺失的接口语义。

## 常见问题

### 为什么页面没有出现在侧边栏？

如果你显式配置了 `pages`，Clarify 不会自动把所有文件插入侧边栏。需要手动添加 `openapi` 条目。

### 为什么构建失败？

通常是 OpenAPI 文件不是合法 JSON/YAML，或 `$ref` 无法解析。先用 OpenAPI 校验工具确认规范有效，再重新构建。

### 可以生成多个 API Reference 吗？

可以。每个 `.openapi.*` 文件都会成为一个独立的 OpenAPI 路由，也可以分别加入侧边栏。
