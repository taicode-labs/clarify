# 插件接口与 UI 插槽设计

本文档记录 Clarify 插件接口的整体扩展方向，并定义第一版运行时 UI 插槽的最小设计。该设计首先服务于基于 GitHub 的评论系统，同时避免把评论能力做成核心特例。

目标是让插件能够扩展 Clarify 的构建流水线、内容处理、运行时 UI 与产物生成能力，同时保持纯静态部署、SSR 与客户端 hydration 一致。

---

## 背景

Clarify 当前插件主要面向构建期扩展，例如路由调整、虚拟模块注入、HTML 转换和构建完成后的产物处理。

但一些能力需要扩展页面运行时 UI，例如：

- GitHub 评论与反馈入口
- 页面底部 CTA
- 相关阅读
- 内容版权声明
- AI 摘要或复制入口
- 内部平台集成信息

这些能力不适合通过 `html:transform` 注入脚本后再操作 DOM，因为这种方式会依赖内部 class 名，无法参与 SSR，且容易造成 hydration 不一致。

因此需要一个更小、更稳定的 UI 扩展点。

---

## 设计原则

1. **核心只提供插槽，不内置业务逻辑**  
   GitHub、评论、分析等能力都应由插件实现。

2. **不修改 route 数据结构作为前提**  
   插槽组件可以通过内置 slot hook 获取当前 route。评论系统可以基于 `route.path` / `route.basePath` 自行生成绑定 key。

3. **上下文通过 hook 获取，而不是组件参数传递**  
   RuntimeSlot 只负责挂载组件。组件需要的 route、locale、config、slot 静态配置等上下文统一通过内置虚拟模块导出的 hook 获取。

4. **构建期注入数据仍然走 hooks**  
   如果插件确实需要改路由、禁用某些页面、读取 frontmatter 或预生成数据，应继续使用 `routes:resolved` 等 hook。

5. **纯静态兼容**  
   插件 UI 通过虚拟模块进入客户端与 SSR bundle，不要求服务端运行时。

6. **SSR 与 hydration 一致**  
   插件组件必须作为 React 树的一部分渲染，而不是在客户端额外操作 DOM。

---

## 非目标

第一版不解决以下问题：

- 不设计通用 route 扩展字段，例如 `route.extensions` / `route.data`
- 不支持无限制布局扩展
- 不支持插件动态新增任意页面区域
- 不设计插件级 i18n 注册系统
- 不设计 OAuth 或服务端代理机制
- 不让插件直接接管核心页面布局

这些能力可以在真实需求出现后再扩展。

## 当前插件接口分层

Clarify 现有插件接口已经覆盖了构建流水线的主要阶段：

```ts
export type ClarifyPlugin = {
  name: string
  hooks: Partial<ClarifyHooks>
}
```

现有 hook 可以按职责分为几层：

| 层级 | Hook | 适合能力 |
|------|------|----------|
| 内容层 | `pages:resolved`、`page:transform` | frontmatter 规范化、内容重写、自动补充页面元信息 |
| 路由层 | `routes:discover`、`routes:discovered`、`routes:resolved` | 新增内容源、过滤页面、改导航、补全路由信息 |
| 模块层 | `modules:before` | 注入虚拟模块、生成运行时配置、提供插件 runtime 入口 |
| HTML 层 | `html:transform` | 注入 meta、script、link、preload 等 HTML shell 资源 |
| 开发层 | `dev:configureServer` | 本地开发服务、调试接口、内容 artifact 预览 |
| 产物层 | `build:done` | sitemap、搜索索引、SEO 文件、额外静态资源 |

因此插件扩展不需要从一个“大而全”的新接口开始。更合理的方向是：

1. 保留 hooks 作为构建期扩展主干。
2. 增加少量声明式能力，解决 hooks 不适合表达的场景。
3. 每个声明式能力都应能被编译成现有流水线里的虚拟模块、HTML 转换或产物生成。

第一版 `slots` 就属于声明式能力：它不是新的业务系统，而是对 `modules:before` + renderer 固定挂载点的结构化封装。

## 可插件化能力地图

从当前系统看，后续可以开放给插件的能力大致分为以下几类。

| 能力 | 当前状态 | 推荐扩展方式 | 示例插件 |
|------|----------|--------------|----------|
| 内容源 | 已有路由发现 hook | 继续增强 hooks | OpenAPI、远程 Markdown、CMS 同步 |
| 页面转换 | 已有 `page:transform` | 保持 hook | 自动摘要、frontmatter 默认值、MDX 内容改写 |
| 路由与导航 | 已有 `routes:*` hooks | 保持 hook，必要时补类型 | 自动分组、隐藏草稿、版本化文档 |
| 运行时 UI | 当前缺少稳定入口 | 新增 `slots` | GitHub 评论、反馈、CTA、相关阅读 |
| 运行时数据 | 可通过虚拟模块实现 | 继续使用 `modules:before`，后续可封装 `runtimeModules` | 评论 key 映射、推荐内容索引 |
| HTML shell | 已有 `html:transform` | 保持 hook | 分析脚本、站点验证、preconnect |
| 静态产物 | 已有 `build:done` | 保持 hook | 搜索索引、llms.txt、RSS、manifest |
| 开发服务 | 已有 `dev:configureServer` | 保持 hook | 本地预览 API、调试面板数据 |
| 主题 | 已有 CSS variables 和配置 | 暂不开放插件主题注册 | 品牌样式包可先通过 CSS/配置解决 |
| i18n | 已有站点 locale | 暂不设计插件文案注册 | 插件内部根据 `locale` 管理文案 |
| MDX 组件 | 当前由 renderer provider 控制 | 后续可考虑 `mdxComponents` 声明 | 自定义 callout、代码块增强 |
| 搜索 | 当前内置插件实现 | 可逐步外置为插件配置 | Pagefind、远程搜索、Algolia |

判断一项能力是否应该插件化，可以用以下规则：

- 如果它改变内容、路由、导航或构建产物，优先用 hook。
- 如果它只需要在页面固定位置渲染 UI，优先用 `slots`。
- 如果它需要把构建期数据传给运行时，优先用虚拟模块。
- 如果它需要影响最终 HTML shell，使用 `html:transform`。
- 如果它需要新的公共 API，必须先证明现有 hook/virtual module 组合表达成本过高。

## 第一版最小接口

### 插槽名称

`UISlotName` 不应围绕单个插件场景命名，而应反映 Clarify 页面结构中稳定、可长期维护的位置。

命名规则是一个点分路径，不强制固定段数。它由三部分语义组成：

```ts
type UISlotName = `${Scope}.${Path}.${Position}`
```

其中 `Path` 可以是一层，也可以是多层，取决于实际 UI 结构深度。

| 部分 | 说明 | 示例 |
|------|------|------|
| `Scope` | 插槽所属结构层级（第一段） | `layout`、`page`、`content` |
| `Path` | 一层或多层稳定 UI 区域，反映真实嵌套结构 | `footer`、`sidebar.navigation`、`openapi.operation` |
| `Position` | 末段，表示相对插入位置或语义位置 | `before`、`after`、`start`、`end`、`actions`、`replace` |

为什么不强制三段：页面结构本身存在深浅不一的嵌套。强行压成三段会导致命名失真，例如侧栏导航底部、OpenAPI operation 标题操作区都需要更深的路径。命名应跟随结构，而不是反过来约束结构。

示例：

```ts
'page.footer.before'            // 三段：页面页脚之前
'layout.sidebar.navigation.after' // 四段：布局侧栏导航之后
'page.openapi.operation.actions'// 四段：OpenAPI operation 标题操作区
'content.mdx.heading.after'     // 四段：MDX 标题之后
```

命名约束：

1. 第一段必须是已知 `Scope`。
2. 最后一段必须是已知 `Position`。
3. 中间段描述稳定的 UI 区域，按从外到内的嵌套顺序书写。
4. 段名使用小写，多词用连字符，例如 `code-group`。

建议的初始命名空间：

```ts
export type UISlotName =
  | 'layout.header.before'
  | 'layout.header.after'
  | 'layout.sidebar.before'
  | 'layout.sidebar.after'
  | 'layout.sidebar.navigation.after'
  | 'page.main.before'
  | 'page.main.after'
  | 'page.title.before'
  | 'page.title.after'
  | 'page.title.actions'
  | 'page.body.before'
  | 'page.body.after'
  | 'page.navigation.before'
  | 'page.navigation.after'
  | 'page.footer.before'
  | 'page.footer.after'
```

分层说明：

| Scope | 用途 |
|-------|------|
| `layout` | 站点外壳级位置，例如 header、sidebar。适合全站公告、辅助入口、侧栏扩展。 |
| `page` | 当前路由页面框架位置，例如 main、title、navigation、footer。适合评论、反馈、页面 CTA。 |
| `content` | 具体内容渲染器内部位置，例如 MDX body 或 OpenAPI operation。只有当对应渲染器明确接入时才开放。 |

第一批建议先实现 renderer 已稳定掌控的位置：

```ts
export type UIEExtensionSlotName =
  | 'page.title.actions'
  | 'page.navigation.before'
  | 'page.navigation.after'
  | 'page.footer.before'
  | 'page.footer.after'
```

其中评论、页面反馈、底部 CTA 应使用：

```ts
'page.footer.before'
```

它位于 `PageNavigation` 之后、`PageFooter` 之前，命名遵循统一的点分路径规则，也方便未来增加 `page.footer.after`、`layout.header.after`、`layout.sidebar.navigation.after` 等更深层位置。

### 插槽声明

插件通过 `slots` 声明运行时 UI：

```ts
export type UISlotRegistration = {
  name: UISlotName
  component: string
  config?: Record<string, unknown>
}
```

字段说明：

| 字段 | 说明 |
|------|------|
| `name` | 插槽名称 |
| `component` | React 组件模块路径，可以是普通模块，也可以是插件注入的 virtual module |
| `config` | 静态配置，构建期序列化后放入 slot context，通过 `useSlot()` 读取 |

第一版不加入 `order`、`clientOnly`、`enabled` 等字段。若同一插槽有多个组件，先按插件数组顺序渲染。

---

### 插件接口

```ts
export type ClarifyPlugin = {
  name: string
  hooks: Partial<ClarifyHooks>
  slots?: UISlotRegistration[]
}
```

`slots` 与 `hooks` 平级，表示运行时 UI 扩展声明。

### 扩展型与替换型插槽

插槽分为两类，它们的语义、冲突规则和默认行为都不同，因此 slot 名称应明确区分：

| 类型 | 命名后缀 | 语义 | 多插件冲突 | 默认行为 |
|------|----------|------|------------|----------|
| 扩展型 | `before` / `after` / `actions` | 在固定位置追加 UI，不影响核心组件 | 允许多个，按顺序渲染 | 无插件时不渲染额外内容 |
| 替换型 | `replace` | 用插件组件替换核心默认组件 | 只允许一个，多个视为冲突 | 无插件时渲染 Clarify 内置默认组件 |

替换型插槽让用户可以替换 Clarify 的内置组件，例如自定义页脚、自定义上一页/下一页导航、自定义内容操作栏。

建议的替换型插槽：

```ts
export type UIReplaceableSlotName =
  | 'page.banner.replace'
  | 'page.footer.replace'
  | 'page.navigation.replace'
  | 'layout.header.replace'
  | 'layout.sidebar.replace'
```

完整 slot 名称类型由两部分组成：

```ts
export type UISlotName =
  | UIEExtensionSlotName
  | UIReplaceableSlotName
```

替换型插槽的关键规则：

1. **独占**：同一个 `*.replace` 插槽只能被一个插件占用。如果多个插件声明同一替换插槽，应在构建期报错，而不是静默覆盖。
2. **有默认回退**：替换型插槽必须对应一个 Clarify 内置默认组件。无插件占用时，渲染默认组件，行为与现在完全一致。
3. **接收默认组件引用**：替换组件可以选择包裹或完全替换默认实现。默认组件通过 slot context 暴露：

```tsx
import { useSlot } from 'virtual:clarify/slot'

export default function CustomFooter() {
  const { DefaultComponent, route } = useSlot()

  return (
    <footer>
      <MyExtraLinks route={route} />
      <DefaultComponent />
    </footer>
  )
}
```

4. **与扩展型分离**：`RuntimeSlot` 处理扩展型插槽（可能渲染多个组件）；替换型插槽由独立的 `ReplaceableSlot` 处理（最多渲染一个组件，否则回退到默认组件）。

这样替换能力不会污染扩展型插槽的多组件渲染模型，也不会让 `RuntimeSlot` 同时承担“追加”和“替换”两种语义。

### 与现有 banner/footer 组件加载的关系

这是一个需要明确的设计决策：Clarify 目前已经支持把 `banner` 和 `footer` 配置为组件路径，CLI 会把它编译进 `virtual:clarify-runtime`，由 `AppShell` 用它替换内置的 `PageBanner` / `PageFooter`。

这套机制与替换型插槽（`page.footer.replace` 等）在能力上是**重复的**——两者都是“用自定义组件替换内置区域组件”。如果两套机制同时存在，会带来几个问题：

- 两条不同的配置入口（`clarify.ts` 的组件路径 vs 插件 `slots` 声明）做同一件事，用户需要理解两套心智模型。
- 当两者同时配置同一区域时，优先级和冲突规则会变得含糊。
- 维护成本翻倍：组件加载、SSR 注入、类型生成都要各做一遍。

因此结论是：**替换型插槽一旦落地，就应该成为唯一的区域替换机制，现有的 banner/footer 组件路径加载应收敛进来，而不是并存。**

推荐的收敛方向：

1. **保留纯配置形态**：`banner` / `footer` 作为“配置对象”（链接、文案、社交项等）的用法保持不变，这部分不是组件替换，不与插槽重叠。
2. **组件形态迁移到替换型插槽**：现在“`banner`/`footer` 接收组件路径”的用法，等价于内置一个占用 `page.banner.replace` / `page.footer.replace` 的官方插件。组件加载、`DefaultComponent` 回退、SSR 注入都复用插槽的同一条通路。
3. **过渡期兼容**：可以在内部把“组件路径配置”实现为自动注册对应替换型插槽的语法糖，对用户保持向后兼容，底层只保留插槽一条实现路径。

这样既不削弱现有能力，也避免在引入插槽后长期维护两套等价机制。

### 内置 Slot Context Hook

运行时插槽组件不接收上下文 props。组件通过 Clarify 内置虚拟模块读取当前 slot 上下文：

```ts
import { useSlot } from 'virtual:clarify/slot'

export type SlotContext = {
  name: UISlotName
  plugin: string
  route?: RouteItem
  locale?: string
  config?: Record<string, unknown>
  DefaultComponent?: ComponentType
}
```

字段说明：

| 字段 | 说明 |
|------|------|
| `name` | 当前插槽名称 |
| `plugin` | 当前 slot definition 所属插件名 |
| `route` | 当前页面路由。包含 `path`、`basePath`、`locale`、`title` 等现有字段 |
| `locale` | 当前语言，例如 `zh-CN` / `en-US` |
| `config` | 插件在 slot 声明中提供的静态配置 |
| `DefaultComponent` | 仅替换型插槽提供，指向 Clarify 内置默认组件，便于插件选择包裹或完全替换 |

这种方式让 slot 组件签名保持普通 React component 形式：

```tsx
export default function MySlot() {
  const slot = useSlot()
  return null
}
```

`RuntimeSlot` 负责为每个组件提供独立 context。组件不需要从 props 接收 `route`、`locale`、`config` 或其他 Clarify 上下文。

### 项目级虚拟模块类型声明

由于 `virtual:clarify/slot`、`virtual:clarify-runtime-slots` 这类模块只在 Clarify/Vite 构建期间存在，用户项目中的 TypeScript 语言服务默认无法识别它们。

为了避免插件作者或站点作者手动编写 `declare module`，Clarify dev 应根据当前项目与启用插件自动生成项目级类型声明文件，例如：

```txt
clarify-env.d.ts
```

该文件可以包含：

```ts
declare module 'virtual:clarify/slot' {
  import type { RouteItem } from '@clarify-labs/renderer'

  export type UISlotName =
    | 'page.title.actions'
    | 'page.navigation.before'
    | 'page.navigation.after'
    | 'page.footer.before'
    | 'page.footer.after'

  export type SlotContext = {
    name: UISlotName
    plugin: string
    route?: RouteItem
    locale?: string
    config?: Record<string, unknown>
  }

  export function useSlot(): SlotContext
}
```

设计要求：

- dev 启动时生成，插件配置变化时更新。
- build 前也应确保生成一次，避免 CI 类型检查缺少声明。
- 文件名使用 `clarify-env.d.ts`，放在用户项目根目录，类似 Vite 的 `vite-env.d.ts`。
- 不覆盖用户自己的 `env.d.ts` 或其他类型声明文件。
- 文件内容应带有 generated 标记，方便用户识别，不鼓励手动编辑。
- 项目模板默认把 `clarify-env.d.ts` 纳入 `tsconfig.json` include；如果现有项目没有包含，dev 可以提示用户加入。
- 插件未来如果声明自定义 virtual module，也应能把对应类型片段贡献到这个文件中。

这样运行时代码通过虚拟模块实现，开发期类型也由 Clarify 自动补齐，用户不会看到 `Cannot find module 'virtual:clarify/slot'` 之类错误。

## 运行时注册模块

CLI 根据所有插件的 `slots` 生成虚拟模块，例如：

```ts
// virtual:clarify-runtime-slots

import Slot0 from 'virtual:clarify-github-comments-slot'

export const runtimeSlots = {
  'page.footer.before': [
    {
      plugin: 'clarify:github-comments',
      component: Slot0,
      config: {
        repository: 'taicode-labs/clarify',
        category: 'Docs Comments',
      },
    },
  ],
}
```

renderer 侧消费该注册表，并在 `AppShell` 固定位置渲染：

```tsx
<PageNavigation navigation={currentNavigation.items} currentRoute={currentRoute} />
<RuntimeSlot name="page.footer.before" />
<PageFooter component={footerComponent} />
```

`RuntimeSlot` 内部从 AppShell 或 renderer context 读取当前 route 与 locale，然后写入 slot context provider。

---

## RuntimeSlot 行为

`RuntimeSlot` 负责：

1. 根据 slot 名称读取注册组件
2. 从 renderer 内部上下文读取当前 `route` 与 `locale`
3. 为每个 slot component 创建独立 `SlotContext`
4. 按注册顺序渲染组件
5. 不处理业务逻辑

伪代码：

```tsx
export function RuntimeSlot(props: { name: UISlotName }) {
  const definitions = runtimeSlots[props.name] ?? []
  const route = useCurrentRoute()
  const locale = useCurrentLocale()

  return definitions.map((definition) => {
    const Component = definition.component

    return (
      <SlotProvider
        key={definition.plugin}
        value={{
          name: props.name,
          plugin: definition.plugin,
          route,
          locale,
          config: definition.config,
        }}
      >
        <Component />
      </SlotProvider>
    )
  })
}
```

`useSlot()` 只能在 slot component 或其子组件内调用。如果组件不在 slot provider 下调用，应抛出明确错误，方便插件作者定位问题。

## GitHub 评论插件示例

插件声明：

```ts
export function githubCommentsPlugin(options: {
  repository: string
  category?: string
}): ClarifyPlugin {
  return {
    name: 'clarify:github-comments',
    hooks: {},
    slots: [
      {
        name: 'page.footer.before',
        component: 'virtual:clarify-github-comments-slot',
        config: options,
      },
    ],
  }
}
```

运行时组件：

```tsx
import { useSlot } from 'virtual:clarify/slot'

export default function GithubCommentsSlot() {
  const { route, locale, config } = useSlot()
  if (!route) return null

  const pathname = route.basePath ?? route.path
  const commentsKey = `pathname:${pathname}`

  return (
    <GithubComments
      commentsKey={commentsKey}
      locale={locale}
      repository={config?.repository as string}
      category={config?.category as string | undefined}
    />
  )
}
```

这样文章与 GitHub Discussion 的绑定关系由插件组件通过 slot hook 读取当前 route 后自行计算，不要求核心修改 route 类型，也不要求 RuntimeSlot 把 route 作为组件参数传入。

---

## 何时使用 hooks

如果插件需要构建期控制，应继续使用 hooks，而不是要求 slot 接口变大。

适合使用 hooks 的场景：

- 读取 frontmatter 中的 `comments: false`
- 某些路由禁用评论
- 自定义某些页面的评论 key
- 根据页面类型决定是否展示 slot
- 构建期生成额外静态数据

例如：

```ts
hooks: {
  'routes:resolved'(input) {
    return {
      ...input,
      routes: input.routes.filter(route => route.path !== '/404'),
      navigation: input.navigation,
    }
  },
}
```

slot 组件只负责运行时渲染；构建期页面处理仍由 hooks 完成。

---

## 与主题的关系

插件 UI 不需要显式接收主题对象。

推荐方式：

```tsx
<section className="rounded-(--clarify-theme-tokens-radius-xl) border border-(--clarify-theme-tokens-colors-border) bg-(--clarify-theme-tokens-colors-surface)">
  ...
</section>
```

如果插件确实需要当前 light / dark 状态，可以使用 renderer 导出的 `useTheme()`。

---

## 与语言的关系

插件组件通过 `useSlot()` 获取当前语言：

```ts
const { locale } = useSlot()
const currentLocale = locale ?? 'en-US'
```

插件自己的文案可以内部维护：

```ts
const messages = {
  'zh-CN': { title: '评论' },
  'en-US': { title: 'Comments' },
}
```

第一版不提供插件级文案注册系统。

---

## 为什么不引入 route.data / route.extensions

GitHub 评论系统的默认绑定 key 可以直接由 route 计算：

```ts
const commentsKey = `pathname:${route.basePath ?? route.path}`
```

因此不需要为了评论系统修改 route 类型。

如果未来某个插件确实需要页面级构建期数据，可以通过 hook 自行调整 route 或通过额外虚拟模块维护映射表，而不是让最小 UI slot 接口承担这个复杂度。

## 插件接口演进路线

### 第一阶段：稳定 UI 插槽

只新增：

- `ClarifyPlugin.slots`
- `page.footer.before`
- `RuntimeSlot`
- `virtual:clarify-runtime-slots`
- `virtual:clarify/slot` 与 `useSlot()`
- dev/build 自动生成项目级 `clarify-env.d.ts`

该阶段解决评论、反馈、底部 CTA 这类最明确的 UI 扩展需求，同时避免 slot component props 随上下文增长而膨胀，并避免虚拟模块在用户项目里产生类型错误。

### 第二阶段：运行时模块声明

如果多个插件都需要通过 `modules:before` 注入运行时数据，可以考虑增加声明式模块接口：

```ts
runtimeModules?: {
  id: string
  code: string | (() => string | Promise<string>)
}[]
```

但在第一版中仍然使用现有 `modules:before`，避免重复抽象。

### 第三阶段：更多稳定插槽

只有当具体插件证明 `page.footer.before` 不够用时，再增加新的 slot，例如：

- `page.afterContent`
- `page.beforeNavigation`
- `page.title.actions`
- `layout.sidebar.footer`

新增 slot 必须满足两个条件：

1. 位置足够稳定，不暴露内部布局细节。
2. 至少有一个明确插件场景无法通过已有 slot 合理实现。

### 第四阶段：替换型插槽

在扩展型插槽稳定后，再开放替换型插槽（`*.replace`），允许插件替换 Clarify 内置默认组件，例如：

- `page.footer.replace`
- `page.navigation.replace`
- `layout.header.replace`

替换型插槽必须满足：

1. 对应一个稳定的内置默认组件，且默认组件通过 slot context 暴露。
2. 同一替换插槽独占，多插件冲突时构建期报错。
3. 替换组件可访问与默认组件相同的上下文，保证 SSR 与 hydration 一致。

### 第五阶段：MDX 与搜索等专项能力

后续可以评估：

- `mdxComponents`：插件注册 MDX component provider 扩展。
- `searchProvider`：允许替换或增强搜索实现。
- `contentSources`：把远程内容源从 route hooks 中抽象出来。
- `artifacts`：把常见静态产物声明式化。

这些都应建立在现有 hooks 使用经验之上，而不是提前设计完整插件平台。

---

## 结论

Clarify 插件接口应保持“hooks 为主干，声明式能力为补充”的结构。

第一版插件 UI 机制应保持极小：

```ts
slots?: {
  name: 'page.footer.before'
  component: string
  config?: Record<string, unknown>
}[]
```

运行时组件不接收 Clarify 上下文参数，而是调用：

```ts
const slot = useSlot()
```

核心只负责提供稳定插槽、React 渲染通道和内置 slot context hook；业务数据绑定、页面过滤和构建期增强仍由插件组件与 hooks 自行处理。
