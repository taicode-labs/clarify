# 产品路线图

Clarify 的路线图围绕一个长期目标展开：提供一个开源、本地优先、可自托管的文档发布工具，把 MDX 内容、OpenAPI Reference、多语言站点、主题系统和 AI-ready 输出统一到同一套工程化工作流中。

这份路线图会优先描述**已经交付的稳定能力**，再列出正在推进和计划探索的方向。未完成事项不承诺具体发布日期，实际优先级会根据用户反馈、维护成本和生态需求调整。

## 近期重点

近期工作会优先围绕“让现有能力更稳定、更完整、更适合真实项目上线”展开。

| 方向 | 目标 |
|------|------|
| 搜索体验打磨 | 优化 Pagefind 加载反馈、结果摘要、高亮、键盘操作和多语言切换后的体验一致性。 |
| SEO 与分享卡片 | 完善 Open Graph、Twitter Card、canonical URL、页面级标题和描述策略。 |
| OpenAPI 体验完善 | 优化复杂 Schema、错误状态、示例选择、代码示例和大型规范的加载表现。 |
| 主题系统打磨 | 将更多视觉规则沉淀为稳定 token，完善主题编辑器导出和文档说明。 |
| 文档与模板 | 完善完整模板、迁移指南、部署指南、内置组件参考和最佳实践。 |
| 测试覆盖 | 增强 CLI、路由、OpenAPI、i18n、主题和渲染器组件的回归测试。 |

## 中期规划

这些能力会在核心体验稳定后逐步推进。

- [ ] **版本化文档**：支持多个文档版本并行发布、切换和归档。
- [ ] **自定义 MDX 组件注册**：提供稳定机制，让项目组件进入 MDX 作用域。
- [ ] **外部搜索适配**：在内置 Pagefind 之外，为 Algolia 或企业内部搜索提供可插拔适配。
- [ ] **内容质量检查**：检查断链、缺失翻译、重复标题、失效 OpenAPI 引用等问题。
- [ ] **发布预览与构建报告**：在构建后输出路由、产物、警告和性能摘要。
- [ ] **更完整的插件 API**：为搜索、翻译、内容治理和内部平台集成提供更稳定的扩展点。

## 长期方向

Clarify 的长期方向不是变成封闭平台，而是成为一个可以被团队二次开发、嵌入内部流程、并长期自托管的开源文档基础设施。

- **文档工程工作台**：围绕内容预览、翻译状态、断链检查、OpenAPI diff 和发布审查提供更完整的工具链。
- **生态插件**：沉淀搜索、评论、分析、认证、文档版本、AI 索引等常用能力插件。
- **编辑器集成**：探索 VS Code 预览、自动补全、配置校验和 OpenAPI 引用跳转。
- **企业内网部署**：改善私有部署、子路径部署、离线资产和安全策略相关体验。
- **AI 文档协议**：继续优化 `llms.txt`、原始内容引用、结构化页面元数据和 agent 读取体验。

## 如何参与

Clarify 会优先采纳能改善真实文档站点体验的反馈。如果你希望推动某项能力：

- 在 [GitHub Issues](https://github.com/taicode-labs/clarify/issues) 提交功能请求或使用场景
- 在 [开发指南](/development) 查看本地开发、代码规范和贡献流程
- 在 Pull Request 中说明问题背景、设计取舍和验证方式

路线图不是固定清单。我们会持续根据社区反馈、维护成本和项目定位调整优先级。
