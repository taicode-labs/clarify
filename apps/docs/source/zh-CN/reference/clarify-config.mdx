# Clarify 配置参考

推荐使用 `clarify.ts` 作为项目配置文件。`clarify.js` 和 `clarify.json` 也受支持，加载优先级为：

```txt
clarify.ts > clarify.js > clarify.json
```

`clarify.json` 适合不需要插件函数的简单项目；需要插件或更强类型提示时使用 `clarify.ts`。

---

## 配置分层

Clarify 将**项目配置**和**运行参数**明确分离：

| 层级 | 受众 | 入口 | 职责 |
|------|------|------|------|
| 项目配置 | 内容作者、技术写作者、扩展开发者 | `clarify.ts/js/json` | 站点展示、多语言、导航、插件 |
| CLI 参数 | 开发者、构建环境 | `clarify dev/build` | 项目根目录、内容目录、输出目录、端口 |

> 内容创作者主要维护 `clarify.ts` 和 `source`；开发者通过 CLI 参数控制运行方式。

## 完整配置类型

```ts
import { defineConfig } from '@clarify-labs/cli'

export default defineConfig({
  title: 'Clarify 文档',
  description: 'Clarify 官方文档 — 开源文档发布工具',
  siteUrl: 'https://docs.example.com',
  routePrefix: '/docs/',
  assetPrefix: 'https://cdn.example.com/docs/',
  logo: {
    light: '/logo-light.svg',
    dark: '/logo-dark.svg',
  },
  homeUrl: 'https://example.com',
  favicon: '/favicon.ico',
  theme: {
    preset: 'default',
    tokens: {
      colors: {
        primary: { light: '#00D492', dark: '#34D399' },
        accent: '#0D9488',
        background: { light: '#ffffff', dark: '#09090b' },
        foreground: { light: '#18181b', dark: '#fafafa' },
        surface: { light: '#f8fafc', dark: '#18181b' },
        muted: { light: '#64748b', dark: '#a1a1aa' },
        border: { light: '#e2e8f0', dark: '#27272a' },

      },
    },

      maxWidth: '88rem',
    },
  },
  navigation: {
    links: [
      { label: { 'zh-CN': '文档', 'en-US': 'Docs' }, href: '/getting-started' },
      { label: 'GitHub', href: 'https://github.com/taicode-labs/clarify', external: true },
    ],

      {

      },
    ],
  },

    product: {
      name: 'Clarify',
      version: '0.8.0',
    },

      support: 'https://clarify.pub/support',

      mcp: true,

    },

    },
```

## 顶层字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `title` | `string` | `"Clarify Docs"` | 站点标题。会出现在侧边栏品牌区、浏览器标题、SEO 标题兜底和站点元数据中。建议使用产品或文档中心名称，而不是页面标题。 |
| `description` | `string` | `""` | 站点级描述。用于 SEO 描述、`llms.txt` 摘要和分享卡片兜底。建议写成一句 80–160 字符的产品说明。 |
| `siteUrl` | `string` | — | 公开访问的站点 origin，必须包含协议且不包含 `routePrefix`。配置后会启用依赖稳定公网绝对地址的 SEO 产物和复制操作。 |
| `routePrefix` | `string` | `"/"` | 站点页面路由的挂载前缀。 |
| `assetPrefix` | `string` | `routePrefix` | 构建产物中静态资源的路径或绝对 URL 前缀。 |
| `logo` | `string \| { light?: string; dark?: string }` | — | 品牌 Logo。可以使用单一路径，也可以按亮色/暗色主题分别提供资源。路径通常放在 `public/` 或内容目录可访问的位置。 |
| `homeUrl` | `string` | `"/"` | 点击站点品牌标识时跳转的地址。 |
| `favicon` | `string \| { light?: string; dark?: string }` | — | 浏览器标签页图标。支持单一路径或亮暗主题变体。建议使用 `.ico`、`.svg` 或尺寸完整的 `.png`。 |
| `theme` | `{ preset?: 'default' \| 'base'; tokens?: ClarifyThemeTokensConfig; layout?: ClarifyThemeLayoutConfig }` | `{ preset: 'default' }` | 主题配置。通过 `preset` 选择基底，通过 `tokens` 和 `layout` 做局部覆盖。 |
| `navigation` | `{ links?: ClarifyNavbarLink[]; tabs?: ClarifyTabsConfig }` | — | 顶部导航链接、站点一级分类和侧边栏结构。 |
| `banner` | `ClarifyBannerConfig` | — | 顶部公告条。适合展示版本发布、迁移提示、维护窗口、活动通知等短消息。 |
| `footer` | `ClarifyFooterConfig` | — | 页脚。可放置常用链接、社区入口、版权声明、外部社交账号，或自定义页脚组件路径。Clarify 品牌标识始终保留。 |
| `locales` | `ClarifyLocalesConfig` | — | 可用语言、默认语言、文本方向和缺失翻译策略。 |
| `variables` | `Record<string, string \| number \| boolean \| object>` | `{}` | 全局项目变量。可在 MDX frontmatter、正文和 OpenAPI JSON/YAML 规范中用 `{{ name }}` 或 `{{ nested.name }}` 占位符复用产品名、版本号、链接、联系方式、API 基础地址和法律声明等内容。 |
| `features` | `ClarifyFeaturesConfig` | 全部开启 | 可选能力及其细节配置。`repository` 描述文档关联的源码仓库。 |
| `plugins` | `ClarifyPlugin[]` | `[]` | 构建期插件列表。只能在 `clarify.ts` 或 `clarify.js` 中使用，不能写入 `clarify.json`。 |

主题覆盖能力还包括 `theme.tokens.colors`、`theme.tokens.radius` 和 `theme.layout.maxWidth`。站点是否暴露内置主题编辑器由 `features.themeEditor` 控制。

## `title` 和 `description`

`title` 是站点名称，不是某个页面的标题。`description` 是站点级简介，会作为 SEO 和 AI 摘要产物的兜底描述。页面级标题和描述仍应写在具体 MDX 内容中。

```ts
export default defineConfig({
  title: 'Acme Docs',
  description: 'Documentation for Acme APIs, SDKs, and platform guides.',
})
```

常见建议：

- `title` 保持短小，通常 2–5 个词。
- `description` 写清楚“给谁看、解决什么问题”。
- 不要把版本号写死在 `title` 中；版本信息更适合放在 `banner` 或导航中。

---

## `siteUrl` 和 `features.repository`

`siteUrl` 表示生产环境的公开 origin，必须包含协议，例如 `https://docs.example.com`。如果文档部署在子路径，只在 `routePrefix` 中配置该子路径；Clarify 会在生成绝对地址时组合两者。

`features.repository` 描述文档源码所属的代码仓库。Clarify 当前用它生成“编辑此页面”链接；这份仓库级配置未来还可以承载更多仓库集成能力。

| 字段 | 必填 | 说明 |
|------|------|------|
| `repository` | 是 | 仓库网页地址，例如 `https://github.com/acme/docs` |
| `branch` | 否 | 源文件所在分支，默认通常按 `main` 处理 |
| `directory` | 否 | 内容目录在仓库中的路径前缀；当 `source/` 不在仓库根目录时需要配置 |

<CodeGroup>

```ts title="生产地址"
export default defineConfig({
  siteUrl: 'https://docs.example.com',
})
```

```ts title="子路径部署"
export default defineConfig({
  siteUrl: 'https://acme.github.io',
  routePrefix: '/docs/',
})
```

```ts title="Git 源仓库"
export default defineConfig({
  features: { repository: {
    url: 'https://github.com/acme/docs',
    branch: 'main',
    directory: 'apps/docs/source',
  } },
})
```

```json title="JSON 配置"
{
  "siteUrl": "https://docs.example.com",
  "features": { "repository": {
    "url": "https://github.com/acme/docs",
    "branch": "main",
    "directory": "source"
  } }
}
```

</CodeGroup>

配置 `siteUrl` 后，`clarify build` 会生成 `sitemap.xml` 和 `robots.txt`，页面也会提供复制原始内容链接、`llms.txt` 链接和 MCP 配置的操作。没有配置时，所有支持相对地址的能力仍然可用，包括页面路由、静态资源、原始内容文件、`llms.txt`、`mcp.json` 和搜索索引；依赖稳定公网绝对 URL 的 SEO 文件和上述复制操作不会生成或显示。MCP 仍可通过手动传入实际站点地址来启动，例如 `clarify mcp https://docs.example.com`。

## `logo` 和 `favicon`

```ts
type ClarifyLogoConfig = string | { light?: string; dark?: string }
type ClarifyFaviconConfig = string | { light?: string; dark?: string }
```

`logo` 影响站点品牌展示，`favicon` 影响浏览器标签页、收藏夹和部分系统分享预览。二者都支持：

- 字符串：亮色和暗色主题使用同一份资源。
- 对象：分别为 `light` 和 `dark` 主题提供不同资源。

<CodeGroup>

```ts title="单一资源"
export default defineConfig({
  logo: '/logo.svg',
  favicon: '/favicon.ico',
})
```

```ts title="亮暗资源"
export default defineConfig({
  logo: {
    light: '/brand/logo-light.svg',
    dark: '/brand/logo-dark.svg',
  },
  favicon: {
    light: '/brand/favicon-light.svg',
    dark: '/brand/favicon-dark.svg',
  },
})
```

</CodeGroup>

如果只配置了其中一个主题变体，另一个主题会尽量回退到可用资源。为了避免暗色模式下 Logo 对比度不足，建议为透明 SVG 分别准备亮色和暗色版本。

## `homeUrl`

`homeUrl` 用来配置点击顶部导航 Logo 时跳转的地址，默认值为 `/`。它可以是站内路径，也可以是完整外部链接。

```ts
export default defineConfig({
  homeUrl: 'https://example.com',
})
```

---

## `routePrefix` 与 `assetPrefix`

`routePrefix` 描述页面路由挂载的子路径。默认值是 `/`，表示站点部署在域名根路径。它不会改变内容文件的组织方式，但会影响生成链接和客户端路由。`assetPrefix` 控制构建产物从哪里加载静态资源，默认继承 `routePrefix`；仅当资源使用其他路径或绝对 CDN URL 时单独配置。

```ts
export default defineConfig({
  siteUrl: 'https://acme.github.io',
  routePrefix: '/docs/',
  assetPrefix: 'https://cdn.example.com/acme-docs/',
})
```

使用建议：

- 部署到 `https://docs.example.com/` 时通常不需要配置。
- 部署到 `https://example.com/docs/` 时配置为 `/docs/`。
- 建议以前后都有 `/` 的形式书写，减少路径拼接歧义。

## `theme`

```ts
theme: {
  preset?: 'default' | 'base'
  tokens?: ClarifyThemeTokensConfig
  layout?: ClarifyThemeLayoutConfig
}
```

`theme` 采用“预设 + 局部覆盖”的模型：先选择 `preset`，再通过 `tokens` 和 `layout` 覆盖少量关键变量。没有配置的字段会继续继承预设默认值。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `preset` | `'default' \| 'base'` | `'default'` | 主题基底。决定默认颜色、圆角和布局宽度。 |
| `tokens.colors` | `ClarifyThemeColorTokensConfig` | 继承 preset | 颜色 token。支持字符串或 `{ light, dark }` 亮暗主题对象。 |
| `tokens.radius` | `{ sm?: string; md?: string; lg?: string; xl?: string }` | 继承 preset | 圆角 token。接受任意 CSS 长度，例如 `0.5rem`、`12px`。 |
| `layout.maxWidth` | `string` | 继承 preset | 文档整体最大宽度，例如 `80rem`、`1280px`。 |

当前提供两个预设：

| 预设 | 主色 | 辅助色 | 默认最大内容宽度 | 适用场景 |
|------|------|--------|------------------|----------|
| `default` | `#047857` | `#0f766e` | `82rem` | 带有绿色品牌感的默认文档主题 |
| `base` | `#18181b` | `#52525b` | `80rem` | 无颜色偏好的黑、白、灰标准主题 |

颜色 token 说明：

| Token | 用途 |
|-------|------|
| `primary` | 链接、激活态、强调按钮、当前导航状态 |
| `accent` | 次级强调、辅助高亮 |
| `background` | 页面背景 |
| `foreground` | 正文主文本 |
| `surface` | 卡片、代码容器外层、浮层等表面背景 |
| `muted` | 次要文本、说明文字、弱化 UI |
| `border` | 边框和分割线 |
| `codeBackground` | 行内代码和代码块背景 |

<CodeGroup title='主题常见配置'>

```ts title="默认预设"
export default defineConfig({
  theme: {
    preset: 'default',
  },
})
```

```ts title="品牌色"
export default defineConfig({
  theme: {
    preset: 'base',
    tokens: {
      colors: {
        primary: '#2563eb',
        accent: '#7c3aed',
      },
    },
  },
})
```

```ts title="完整覆盖"
export default defineConfig({
  theme: {
    tokens: {
      colors: {
        primary: { light: '#00D492', dark: '#34D399' },
        background: { light: '#ffffff', dark: '#09090b' },
        foreground: { light: '#18181b', dark: '#fafafa' },
        surface: { light: '#f8fafc', dark: '#18181b' },
        muted: { light: '#64748b', dark: '#a1a1aa' },
        border: { light: '#e2e8f0', dark: '#27272a' },
        codeBackground: { light: '#0f172a', dark: '#020617' },
      },
      radius: {
        sm: '0.375rem',
        md: '0.75rem',
        lg: '1rem',
        xl: '1.5rem',
      },
    },
    layout: {
      maxWidth: '88rem',
    },
  },
  features: {
    themeEditor: true,
  },
})
```

</CodeGroup>

颜色 token 可以直接写字符串，也可以按亮暗模式分别提供 `{ light?: string; dark?: string }`。如果某个模式缺失，Clarify 会回退到可用值或 preset 默认值。当前版本不会读取用户项目中的自定义 CSS 或 `tailwind.config.js` 来覆盖渲染器样式。

## `features`

```ts
type ClarifyFeatureConfig<Options> =
  | boolean
  | ({ enabled?: boolean } & Options)

type ClarifyFeaturesConfig = {
  search?: ClarifyFeatureConfig<{ mcp?: boolean }>
  repository?: ClarifyFeatureConfig<{
    url?: string
    branch?: string
    directory?: string
  }>
  themeEditor?: ClarifyFeatureConfig<Record<never, never>>
  openapi?: ClarifyFeatureConfig<{
    playground?: boolean
  }>
}
```

除生产环境的主题编辑器外，其他内置功能默认启用。每项都支持布尔简写：`false` 禁用整项，`true` 使用默认选项；需要细粒度控制时使用带 `enabled` 的对象。

| 字段 | 默认解析结果 | 说明 |
|------|--------------|------|
| `search` | `{ enabled: true, mcp: true }` | 站内全文搜索（基于 Pagefind）。设置 `mcp: false` 可关闭 MCP 搜索索引与 `mcp.json` 清单的生成。客户端接入和构建产物参见 [MCP 文档搜索](/features/mcp)。 |
| `repository` | `{ enabled: true }` | 文档源码关联的代码仓库。配置 `url` 后可启用页面编辑链接等仓库能力；`branch` 默认使用 `main`，`directory` 对应仓库内的内容目录。 |
| `themeEditor` | `{ enabled: false }` | 内置主题编辑器。开发服务器始终提供编辑器；生产站点需显式设为 `true` 才会展示。 |
| `openapi` | `{ enabled: true, playground: true }` | OpenAPI 页面及请求工作台。响应预览和下载是页面内置能力。 |

```ts title="按需配置功能"
export default defineConfig({
  features: {
    search: true,
    repository: {
      url: 'https://github.com/acme/docs',
      branch: 'main',
      directory: 'docs/source',
    },
    themeEditor: false,
    openapi: {
      playground: true,
    },
  },
})
```

Clarify 始终输出可直接读取的 Markdown/OpenAPI 源文件和 `llms.txt`。配置 `siteUrl` 后，还会输出 `sitemap.xml` 和 `robots.txt`。这些核心构建产物不需要 feature 配置。

`enabled: false` 会禁用整项，即使同一对象中仍填写了子选项也不会生效。

## `variables`

`variables` 用来定义 MDX frontmatter、页面正文和 OpenAPI JSON/YAML 规范中可复用的项目级常量。变量会在 MDX 编译前或 OpenAPI 解析前展开，因此解析后的文本也会用于页面标题、描述、正文标题、章节锚点、接口元数据、搜索索引和复制内容产物。

```ts
export default defineConfig({
  variables: {
    product: {
      name: 'Acme Cloud',
      version: '2.4.0',
    },
    links: {
      support: 'https://support.example.com',
      status: 'https://status.example.com',
    },
    apiBaseUrl: 'https://api.example.com/v1',
  },
})
```

在 MDX 中使用双花括号引用变量：

```mdx
---
title: "{{ product.name }} {{ product.version }}"
description: "Support docs for {{ product.name }}"
---

# Welcome to {{ product.name }}

Current version: {{ product.version }}

- Support: {{ links.support }}
- API base URL: `{{ apiBaseUrl }}`
```

同样的占位符也可以写在 OpenAPI 文件中，Clarify 会在解析和 dereference 规范前先展开变量：

```yaml title="source/api.openapi.yaml"
openapi: 3.0.0
info:
  title: "{{ product.name }} API"
  version: "{{ product.version }}"
servers:
  - url: "{{ apiBaseUrl }}"
```

在 YAML frontmatter 中使用变量时，请给占位符加引号，确保 YAML 将它解析为字符串。

支持的值包括字符串、数字、布尔值和嵌套对象。占位符可以通过点路径引用嵌套字段。未知占位符和对象值占位符会保持原样，因此文档中展示 `{{ user_supplied_value }}` 这类模板示例也是安全的。字符串变量可以继续引用其他变量；循环引用会让构建失败，并输出清晰的错误信息。

## 自定义 404 页面

创建 `source/404.mdx` 即可自定义未找到页面：

```mdx
# 页面未找到

你访问的页面不存在或已经移动。

- [快速开始](/getting-started)
- [配置参考](/reference/clarify-config)
```

Clarify 会在 `/404` 预览这个页面，并在客户端未知路径时复用它作为 404 展示。静态生成时还会额外写出 `output/404.html`，自动文件树导航会默认隐藏 `404`。

## `navigation.links`

```ts
type ClarifyNavbarLink = {
  label: ClarifyLocalizedText
  href: string
  external?: boolean
}
```

`navigation.links` 是站点顶部的全局入口。每个链接包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| `label` | 是 | 链接显示文本，支持本地化文本。 |
| `href` | 是 | 目标地址。内部链接建议使用以 `/` 开头的站内路径；外部链接使用完整 URL。 |
| `external` | 否 | 是否按外部链接处理。配置为 `true` 时通常会以新标签页打开，并带外部链接语义。 |

<CodeGroup>

```ts title="站内链接"
export default defineConfig({
  navigation: {
    links: [
      { label: '文档', href: '/getting-started' },
      { label: 'API', href: '/features/openapi' },
    ],
  },
})
```

```ts title="本地化链接"
export default defineConfig({
  navigation: {
    links: [
      { label: { 'zh-CN': '指南', 'en-US': 'Guides' }, href: '/guides' },
      { label: { 'zh-CN': '更新日志', 'en-US': 'Changelog' }, href: '/changelog' },
      { label: 'GitHub', href: 'https://github.com/taicode-labs/clarify', external: true },
    ],
  },
})
```

```json title="JSON 配置"
{
  "navigation": {
    "links": [
      { "label": "Docs", "href": "/getting-started" },
      { "label": "GitHub", "href": "https://github.com/acme/docs", "external": true }
    ]
  }
}
```

</CodeGroup>

建议顶部导航只保留核心入口，完整信息架构交给侧边栏承载。不要把每个文档章节都放入 `navigation.links`，否则移动端和窄屏体验会变差。

```ts
type ClarifyBannerLink = {
  label: ClarifyLocalizedText

  content: ClarifyLocalizedText
  dismissible?: boolean
  link?: ClarifyBannerLink
}

type ClarifyBannerConfig = ClarifyBannerConfigOptions
```

`banner` 用于顶部公告，适合短消息而不是长段落。

```ts title="内置公告配置"
export default defineConfig({
  banner: {
    content: {
      'zh-CN': '我们正在迁移到新的 API 文档结构',
      'en-US': 'We are migrating to a new API documentation structure',
    },
    link: {
      label: {
        'zh-CN': '查看计划',
        'en-US': 'View plan',
      },

  },
})
```

> 如需完全自定义横幅组件，请使用 [插件机制](/features/plugins) 注册 `page.banner.replace` 插槽。

使用建议：

- 发布、维护、迁移等时效性信息适合放在 `banner`。
- 长期存在的产品说明更适合放在首页或指南页。
- 重要事故或维护窗口建议设置 `dismissible: false`。

## `locales`

```ts
type ClarifyLocalesConfig = {
  default?: string
  missing?: 'fallback' | '404' | 'hide'
  locales: Array<{
    code: string
    label: string
    dir?: 'ltr' | 'rtl'
  }>
}
```

`locales` 开启原生多语言。启用后，内容通常放在 `source/{locale}/` 下，例如 `source/zh-CN/index.mdx` 和 `source/en-US/index.mdx`。

| 字段 | 必填 | 说明 |
|------|------|------|
| `locales` | 是 | 支持的语言列表。每个 `code` 必须唯一。 |
| `default` | 否 | 默认语言。必须是 `locales` 中的某个 `code`；省略时使用第一个 locale。 |
| `missing` | 否 | 缺失翻译策略，默认值为 `fallback`。 |
| `locales[].code` | 是 | 语言代码，也是内容目录名和非默认语言路由前缀。 |
| `locales[].label` | 是 | 语言选择器显示名称。 |
| `locales[].dir` | 否 | 文本方向，`ltr` 或 `rtl`。 |

`default` 对应不带语言前缀的默认路由；非默认语言会带上 locale 前缀。

`missing` 控制某个语言缺少页面时的处理策略：

| 策略 | 行为 | 适合场景 |
|------|------|----------|
| `fallback` | 使用默认语言内容补齐缺失页面 | 先发布主语言，再逐步补翻译 |
| `hide` | 不生成缺失语言页面 | 未翻译内容不希望出现在目标语言导航中 |
| `404` | 缺失页面保持不可访问 | 严格多语言发布，不允许 fallback 内容混入 |

<CodeGroup>

```ts title="缺失时回退"
export default defineConfig({
  locales: {
    default: 'zh-CN',
    missing: 'fallback',
    locales: [
      { code: 'zh-CN', label: '简体中文' },
      { code: 'en-US', label: 'English' },
    ],
  },
})
```

```ts title="隐藏缺失页面"
export default defineConfig({
  locales: {
    default: 'en-US',
    missing: 'hide',
    locales: [
      { code: 'en-US', label: 'English', dir: 'ltr' },
      { code: 'ar', label: 'العربية', dir: 'rtl' },
    ],
  },
})
```

```json title="严格 404"
{
  "locales": {
    "default": "en-US",
    "missing": "404",
    "locales": [
      { "code": "en-US", "label": "English", "dir": "ltr" },
      { "code": "ja-JP", "label": "日本語", "dir": "ltr" }
    ]
  }
}
```

</CodeGroup>

`dir` 会写入生成 HTML 的 `dir` 属性，适合阿拉伯语、希伯来语等 RTL 语言。更多说明见 [配置站点](/guides/navigation)。

## `navigation.tabs` 导航与侧边栏

Clarify 使用 `navigation.tabs` 做站点的一层业务分类。每个 tab 可以配置自己的侧边栏 `pages`。

```ts
type ClarifyTabsConfig = ClarifyTabItem[]

type ClarifyTabItem = {
  tab: ClarifyLocalizedText
  icon?: string
  pages?: ClarifyPagesConfig
}

type ClarifyPagesConfig = ClarifyPagesGroup[] | 'FileTree'
```

如果省略 `navigation.tabs`，Clarify 会根据内容目录的文件结构自动生成一个普通侧边栏，包含 `.mdx` 和 `.openapi.json/.yaml/.yml` 文件。

<CodeGroup title='tabs 常见配置'>

```ts title="文件树 tab"
export default defineConfig({
  navigation: {
    tabs: [
      {
        tab: '文档',
        icon: 'BookOpen',
        pages: 'FileTree',
      },
    ],
  },
})
```

```ts title="手动侧边栏"
export default defineConfig({
  navigation: {
    tabs: [
    {
      tab: { 'zh-CN': '指南', 'en-US': 'Guides' },
      icon: 'Rocket',
      pages: [
        {
          group: { 'zh-CN': '核心路径', 'en-US': 'Core Path' },
          icon: 'Rocket',
          pages: [
            { page: 'getting-started', title: '快速开始', icon: 'Rocket' },
            { page: 'guides/writing-content', title: '写作文档', icon: 'PenLine' },
            { page: 'guides/navigation', title: '配置站点', icon: 'SlidersHorizontal' },
          ],
        },
      ],
    },
    {
      tab: 'API',
      icon: 'BookOpenCheck',
      pages: [
        {
          group: 'API 文档',
          pages: [
            { openapi: 'api.openapi.json', title: 'API Reference', icon: 'BookOpenCheck' },
          ],
        },
      ],
    },
    ],
  },
})
```

```json title="JSON 配置"
{
  "navigation": {
    "tabs": [
      {
        "tab": "Docs",
        "icon": "BookOpen",
        "pages": "FileTree"
      }
    ]
  }
}
```

</CodeGroup>

### `ClarifyTabItem`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `tab` | `ClarifyLocalizedText` | 是 | tab 显示名称。单语言站点可直接写字符串，多语言站点可写本地化文本。 |
| `icon` | `string` | 否 | Lucide 图标名，显示在 tab 名称旁边。 |
| `pages` | `ClarifyPagesConfig` | 否 | 当前 tab 下的侧边栏结构；省略时按文件树生成。 |

最小 tab 配置：

```ts
{
  tab: '文档',
  pages: 'FileTree',
}
```

带本地化名称和图标的 tab：

```ts
{
  tab: { 'zh-CN': '指南', 'en-US': 'Guides' },
  icon: 'Rocket',
  pages: 'FileTree',
}
```

### `ClarifyPagesGroup`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `group` | `ClarifyLocalizedText` | 是 | 侧边栏分组标题。 |
| `icon` | `string` | 否 | Lucide 图标名，显示在分组标题旁边。 |
| `pages` | `ClarifyPagesItem[]` | 是 | 分组下的页面列表，按配置顺序展示。 |

当一个 tab 下有多个信息区块时，可以使用分组：

```ts
{
  group: '快速开始',
  icon: 'Rocket',
  pages: ['index', 'getting-started', 'guides/writing-content'],
}
```

多语言站点可以为分组标题和页面标题使用本地化文本：

```ts
{
  group: { 'zh-CN': '核心路径', 'en-US': 'Core Path' },
  pages: [
    { page: 'getting-started', title: { 'zh-CN': '快速开始', 'en-US': 'Quick start' } },
  ],
}
```

### `ClarifyPagesItem`

| 形态 | 说明 |
|------|------|
| `"getting-started"` | 简写，引用 MDX 页面；当文件自身标题足够清晰时使用。 |
| `{ page }` | 引用 MDX 页面，不写 `.mdx` 后缀；适合需要对象写法时使用。 |
| `{ page, path, title, icon, redirect }` | 引用页面，并可覆盖路由路径、标题、图标或点击目标路径。 |
| `{ openapi, path, title, icon, filter }` | 引用 OpenAPI 规范文件；`path` 可覆盖生成路由，`filter.tags` 可按 operation tags 过滤接口。 |

<CodeGroup title='pages 常见写法'>

```ts title="简写"
'getting-started'
```

```ts title="页面对象"
{
  page: 'guides/writing-content',
}
```

```ts title="自定义标题"
{
  page: 'guides/writing-content',
  title: '写作文档',
  icon: 'PenLine',
}
```

```ts title="显式页面路径"
{
  page: 'guides/writing-content',
  path: 'docs/writing',
  title: '写作文档',
  icon: 'PenLine',
}
```

```ts title="重定向项"
{
  page: 'guides',
  title: '指南',
  icon: 'BookOpen',
  redirect: '/guides/writing-content',
}
```

```ts title="OpenAPI 过滤示例"
{
  openapi: 'api.openapi.json',
  path: 'api/projects',
  title: '示例：Projects API',
  icon: 'BookOpenCheck',
  filter: {
    tags: ['Projects'],
  },
}
```

</CodeGroup>

`path` 会设置 router 实际消费的路由。它是可选的：省略时，Clarify 会根据页面或 OpenAPI 文件名自动生成路由。例如 `page: 'guides/writing-content'` 会生成 `/guides/writing-content`，`openapi: 'api.openapi.json'` 会生成 `/api`。当你希望 URL 与源文件路径不同时，可以使用 `path` 保持稳定路由。

`redirect` 只改变导航项点击后的目标路径，不会创建新的内容页面。它适合把一个导航入口指向更具体的章节、迁移后的路径，或在分组首页不存在时避免用户进入空页面。

OpenAPI 页面默认展示规范中的全部接口。配置 `filter.tags` 后，页面只展示 `operation.tags` 中至少命中一个指定 tag 的接口，并基于 tag 生成独立路由。OpenAPI tag 页面路由会被转成小写 kebab-case。例如 `['Projects']` 会生成 `/api/projects`；`['Admin Users', 'Billing']` 会生成 `/api/admin-users-billing`。设置 `path` 时会覆盖这个生成路由，所以上面的示例会发布到 `/api/projects`，因为显式配置了 `path: 'api/projects'`。启用 i18n 时，默认语言保留配置路由，非默认语言会加上 locale 前缀，例如 `/en-US/api/projects`。

## `plugins`

```ts
plugins: ClarifyPlugin[]
```

`plugins` 是构建期扩展点，可用于调整路由发现、转换 HTML、注入标签或集成外部系统。插件是函数或对象，因此只能在 `clarify.ts` 或 `clarify.js` 中配置，不能写入 `clarify.json`。

```ts
import { defineConfig } from '@clarify-labs/cli'
import { myClarifyPlugin } from './plugins/my-clarify-plugin'

export default defineConfig({
  plugins: [myClarifyPlugin()],
})
```

更多说明见 [插件机制](/features/plugins) 和 [插件 API 参考](/reference/plugin-api)。

## CLI 覆盖项

CLI 参数属于运行时配置，不建议写入 `clarify.ts`。它们用于同一份项目配置在本地、预览环境和 CI 中以不同方式运行。

| CLI 选项 | 内部字段 | 默认值 | 命令 | 说明 |
|----------|----------|--------|------|------|
| `--root <dir>` | `projectRoot` | 当前目录 | `dev` / `build` / `init` | 项目根目录，通常是包含 `clarify.ts` 和 `source/` 的目录。 |
| `--content <dir>` | `rootDirectory` | `source` | `dev` / `build` / `init` | 内容目录，存放 MDX、OpenAPI 和多语言子目录。 |
| `--output <dir>` | `outputDirectory` | `output` | `dev` / `build` / `init` | 构建产物目录。CI 中常覆盖为部署平台要求的目录。 |
| `--host [host]` | Vite dev server host | — | `dev` | 开发服务器监听地址。局域网调试时可设置为 `0.0.0.0`。 |
| `--port <port>` | Vite dev server port | — | `dev` | 开发服务器端口。 |
| `--open [path]` | Vite dev server open | — | `dev` | 启动开发服务器后自动打开浏览器，可指定打开路径。 |

<CodeGroup>

```bash title="默认开发"
clarify dev
```

```bash title="自定义开发"
clarify dev --content docs --port 5173 --open /getting-started
```

```bash title="自定义输出"
clarify build --content source --output dist
```

```bash title="应用根目录"
clarify build --root apps/docs --output output
```

</CodeGroup>

优先级为：

```txt
CLI 参数 > clarify.ts/js/json > 默认值
```

这允许 CI 在不修改项目配置的情况下覆盖输出目录或内容目录。
