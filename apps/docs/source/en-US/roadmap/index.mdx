# Product Roadmap

Clarify's roadmap is guided by one long-term goal: provide an open-source, local-first, self-hostable documentation publishing tool that brings MDX content, OpenAPI references, multilingual sites, theming, and AI-ready output into one engineering workflow.

This roadmap starts with the **stable capabilities that are already delivered**, then outlines the areas currently being prioritized and the directions planned for future exploration. Planned items do not imply a committed release date; priorities may change based on user feedback, maintenance cost, and ecosystem needs.

## Released capabilities

- [x] MDX compilation and rendering
- [x] Frontmatter metadata

- [x] Client hydration and in-site navigation

- [x] Parse `.openapi.json`, `.openapi.yaml`, and `.openapi.yml`
- [x] Generate complete OpenAPI reference pages
- [x] Generate focused API pages with tag filters
- [x] Embed any individual endpoint inside MDX
- [x] Display parameters, request bodies, responses, schemas, examples, authentication, and server information
- [x] Generate copyable request code in multiple languages
- [x] Validate OpenAPI references and operation targets at build time

### Site configuration and information architecture

- [x] TypeScript configuration file (`clarify.ts`)
- [x] Tabs, grouped sidebars, top navigation, footer links, and social links
- [x] External links, redirects, icons, and localized titles
- [x] `routePrefix` support for subpath deployments
- [x] Global announcement banner with localized copy, links, and dismissible state
- [x] Source repository configuration and “Edit this page” links
- [x] Favicon, logo, and site metadata configuration

### Internationalization and theming

- [x] Locale-based content directories

- [x] Missing-translation fallback strategy
- [x] Internal links preserve the current locale context
- [x] Search indexes load for the current language to avoid cross-language result mixing
- [x] Light/dark themes and system preference support
- [x] Theme presets, color tokens, radius tokens, and layout width
- [x] Optional built-in theme editor

### Publishing, AI, and extension points

- [x] Pure static build output
- [x] Public asset copying
- [x] Raw Markdown and OpenAPI content artifacts
- [x] Page copy actions, raw content links, and `llms.txt`
- [x] Pagefind static search indexes and dev-time temporary index serving
- [x] Sitemap, robots, and baseline SEO artifacts
- [x] Plugin hooks: `routes:resolved`, `modules:before`, `html:transform`, `build:assets`, and `build:done`
- [x] Runtime UI slots for plugin-rendered page surfaces, including banner replacement
- [x] `clarify dev`, `clarify build`, and `clarify init`

| Direction | Goal |
|-----------|------|

| SEO and social cards | Improve Open Graph, Twitter Card, canonical URL, page title, and page description behavior. |

| Test coverage | Expand regression coverage around runtime slots, multilingual OpenAPI behavior, themes, and renderer components. |

## Mid-term plans

These capabilities will be explored after the core experience becomes more stable.

- [ ] **Versioned documentation**: publish, switch, and archive multiple documentation versions in parallel.
- [ ] **GitHub-based comments and feedback**: link documentation pages to GitHub Issues or Discussions so readers can leave comments, report problems, request missing content, and let maintainers follow up inside their existing GitHub workflow.
- [ ] **Global content variables**: define reusable project-wide constants for MDX content, such as product names, version labels, support links, contact emails, API base URLs, and legal notices, so fixed content can be updated in one place instead of being edited across many pages.
- [ ] **Custom MDX component registration**: provide a stable way to expose project-specific components to MDX.
- [ ] **Content quality checks**: detect broken links, missing translations, duplicate headings, invalid OpenAPI references, and related issues.
- [ ] **Runtime UI slot expansion**: runtime UI slots have landed for key page surfaces; the next step is to stabilize more slot positions, conflict rules, examples, and type generation for real plugins.
- [ ] **Expanded plugin API**: provide more stable extension points for search, translation, governance, asset emission, and internal platform integrations.

---

## Long-term direction

Clarify's long-term direction is not to become a closed platform. It should become open-source documentation infrastructure that teams can extend, embed into internal workflows, and self-host for the long term.

- **Documentation engineering workspace**: better tooling for content preview, translation status, broken-link checks, OpenAPI diffs, and release review.
- **Ecosystem plugins**: reusable plugins for search, comments, analytics, authentication, documentation versions, and AI indexes.
- **Editor integrations**: explore VS Code preview, autocomplete, config validation, and OpenAPI reference navigation.
- **Private-network deployments**: improve private hosting, subpath deployments, offline assets, and security-policy compatibility.
- **AI documentation protocols**: continue improving `llms.txt`, raw content references, structured page metadata, and agent readability.

## How to participate

Clarify prioritizes feedback that improves real documentation-site workflows. If you want to push a capability forward:

- Submit a feature request or use case in [GitHub Issues](https://github.com/taicode-labs/clarify/issues)
- Read the [Development guide](/development) for local development, code conventions, and contribution workflow
- In pull requests, explain the problem background, design trade-offs, and validation steps

The roadmap is not a fixed checklist. Priorities will continue to evolve with community feedback, maintenance cost, and the project's positioning.
