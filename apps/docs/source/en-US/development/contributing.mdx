# Contributing

Thanks for contributing to Clarify. Contributions should keep the CLI/renderer boundary clear and preserve the static-first design.

---

## Development setup

```bash
git clone https://github.com/taicode-labs/clarify.git
cd clarify
pnpm install
pnpm dev:docs
pnpm dev:www
```

Use `pnpm` for all package operations. Do not use `npm` or `yarn` in this repository.

---

## Development conventions

- Use `source/` as the source directory name.
- Write TypeScript with strict types.
- Prefer explicit types over `any`.
- Keep packages tree-shakeable and application-independent.
- Prefer small, focused changes.
- Package builds use Vite library builds; apps use Vite app builds.
- User-facing docs should describe `clarify dev` and `clarify build`, not internal Vite setup.
- Write commit messages in English.

---

## Choosing where to make a change

| Change type | Usually belongs in |
|-------------|--------------------|
| Config loading, route discovery, plugin hooks, SSG, raw artifacts | `packages/cli` |
| Header, sidebar, MDX components, OpenAPI UI, content actions, theme behavior | `packages/renderer` |
| Official documentation content | `apps/docs/source` |
| Marketing copy and landing pages | `apps/www/source` |

If a feature needs both CLI and renderer changes, define the data shape in the CLI first, expose it through virtual modules, then consume it in the renderer.

---

## Validation commands

Run the checks relevant to your change:

```bash
pnpm typecheck
pnpm build
```

For focused work, use workspace filters:

```bash
pnpm --filter @clarify-labs/cli typecheck
pnpm --filter @clarify-labs/renderer typecheck
pnpm --filter @clarify-labs/docs build
```

Documentation-only changes should still be validated with the docs build when possible because MDX content is compiled.

---

## Commit and PR expectations

All commit messages must be written in English. Conventional Commits are preferred:

| Type | Meaning |
|------|---------|
| `feat:` | New feature |
| `fix:` | Bug fix |
| `docs:` | Documentation-only change |
| `refactor:` | Code change that neither fixes a bug nor adds a feature |
| `chore:` | Tooling, dependency, or maintenance change |

Before opening a PR:

1. keep the change focused;
2. update docs for user-visible behavior;
3. mention affected packages;
4. include validation results;
5. explain design tradeoffs when changing CLI/renderer boundaries.
