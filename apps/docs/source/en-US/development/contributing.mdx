# Contributing

Thank you for your interest in Clarify! The following guide explains how to contribute.

---

## Development environment

```bash
# Clone the repository
git clone https://github.com/taicode-labs/clarify.git
cd clarify

# Install dependencies
pnpm install

# Start development servers
pnpm dev:docs    # Documentation site
pnpm dev:www     # Marketing site
```

---

## Code conventions

### TypeScript

- All code must be TypeScript with `strict: true`.
- Prefer explicit types and avoid `any`.
- Keep packages tree-shakeable (avoid side-effect imports).

### Directory conventions

- Use `source/` as the source directory, not `src/`.
- Packages currently use Vite library build to output ESM / CJS / DTS.
- Apps use Vite + `@tailwindcss/vite` + `@vitejs/plugin-react`.
- User-facing documentation should describe `clarify dev` / `clarify build`; do not require users to maintain internal Vite configuration.

---

## Commit conventions

**All commit messages must be written in English.** Chinese or other languages are not allowed.

### Commit format

Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

| Type | Description |
|------|-------------|
| `feat:` | New feature |
| `fix:` | Bug fix |
| `docs:` | Documentation-only changes |
| `refactor:` | Code change that neither fixes a bug nor adds a feature |
| `chore:` | Build process, tooling, or dependency updates |

### Commit examples

```bash
feat: add sidebar navigation component

- Add NavTree component with recursive rendering
- Support collapsible sections
- Integrate with virtual:clarify/routes

fix: resolve hydration mismatch on 404 pages

docs: update deployment guide with Vercel examples
```

- Keep the subject line within 72 characters.
- Add context in the body when needed.

---

## PR process

1. **Fork the repository** and create a feature branch from `main`.
2. **Write code** and make sure type checks pass.
3. **Update documentation** if you changed user-visible behavior.
4. **Open a PR** with a clear description of what changed and why.

---

## How to choose where to change

| Change type | Usually changed in |
|-------------|--------------------|
| Configuration loading, route discovery, plugin hooks, SSG, raw content artifacts | `packages/cli` |
| Header, sidebar, MDX components, OpenAPI UI, content actions, theme behavior | `packages/renderer` |
| Official documentation content | `apps/docs/source` |
| Marketing copy and landing pages | `apps/www/source` |

Features that span CLI and Renderer should define the build-time data shape first, expose it through virtual modules, and then consume it in the Renderer.

---

## Validation commands

Run commands according to the scope of your changes:

```bash
pnpm typecheck
pnpm build
```

When focusing on a single package, you can use:

```bash
pnpm --filter @clarify-labs/cli typecheck
pnpm --filter @clarify-labs/renderer typecheck
pnpm --filter @clarify-labs/docs build
```

Documentation changes should also run the docs build whenever possible, because MDX content participates in compilation.

---

## Dependency rules

- Use `pnpm`; do not use `npm` or `yarn`.
- Use `workspace:*` for cross-package dependencies.
- Prefer well-maintained, lightweight libraries.
- Limit dependencies to the React ecosystem.
