# Static generation pipeline in detail

> **Scope**: Supports MDX / Markdown and OpenAPI content, outputs an independent HTML file for each route, and hydrates into a React Router app in the browser. Vite is an internal CLI build tool and is not exposed as a user configuration entry.

---

## Goals

| # | Requirement | Implementation |
|---|-------------|----------------|
| 1 | Static deployment: each page has an independent HTML file | Build-time SSR writes `output/<route>/index.html` for each route |
| 2 | Smooth client experience after first paint | Inject `<script>` to hydrate static markup into a React Router SPA |
| 3 | Support MDX / Markdown and OpenAPI rendering | The CLI internal engine scans `*.md`, `*.mdx`, and `*.openapi.*` under `rootDirectory`, then compiles with `@mdx-js/rollup` |
| 4 | Use Vite as the internal build pipeline | Use only public APIs; regular users do not need to maintain Vite configuration |
| 5 | Build-time SSR | Generate a temporary SSR entry, output it to the `.ssr/` subdirectory in the build directory, then import that bundle to render routes |
| 6 | Built-in renderer style pipeline | The CLI internally assembles renderer styles and Tailwind CSS support |

---

## Architecture overview

```
┌─────────────────────────────────────────────────────────────┐
│  Build phase (vite build)                                   │
│                                                             │
│  1. Vite bundles client entry (virtual:clarify/entry-client)│
│       → output/assets/                                      │
│                                                             │
│  2. clarifyPlugin closeBundle():                            │
│       a. Reuse resolved MDX/OpenAPI routes and virtual modules│
│       b. Create temporary SSR entry file                     │
│       c. Vite builds SSR bundle → output/.ssr/entry-server.js│
│       d. Import render(url) from the SSR bundle              │
│       e. For each route: renderToHTML(url) → write HTML      │
│       f. Write raw content and llms.txt                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Runtime (browser)                                           │
│                                                             │
│  1. Browser receives static HTML (SEO-friendly)              │
│  2. `<script>` loads the client bundle                       │
│  3. hydrateRoot mounts React Router on existing markup       │
│  4. Subsequent navigation → client-side SPA navigation       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Key design decisions

### Two-phase build (client + SSR)

Clarify does not require a server runtime. Production builds execute SSR only during the build phase and write the results as static HTML.

To render React components into HTML strings at build time, the CLI generates a temporary SSR entry and then uses Vite to build an SSR bundle:

- **Client bundle**: Normal Vite `build()` outputs JS/CSS assets to `output/` (or the directory specified by `--output`).
- **SSR bundle**: The `closeBundle` phase generates `.ssr/entry-server.js`, then imports and executes it in Node.js.
- **HTML writing**: Reads the `index.html` template produced by the client build and injects `renderToHTML(url)` into `#root`.

This gives us:
- ✅ Artifacts are still directly deployable static files.
- ✅ No server runtime requirement.
- ✅ SSR and client use the same route and virtual module data.

### Virtual modules

Clarify uses virtual modules for clearer separation:

| Virtual module | Purpose |
|----------------|---------|
| `virtual:clarify/config` | Exposes merged project configuration and build options as runtime configuration |
| `virtual:clarify/routes` | Route manifest, navigation tree, and page section data |
| `virtual:clarify/openapi` | Parsed OpenAPI specification registry |
| `virtual:clarify/entry-client` | Client entry injected by the CLI |
| `virtual:clarify-page/*` | Virtual module for each content page |

These modules are resolved by `clarifyPlugin.resolveId()` and generated by `clarifyPlugin.load()`.

### Route discovery

Clarify uses **file-system-based routing** and supports optional frontmatter overrides:

<FileTree aria-label="File-system-based routing">
  <FileTreeItem name="rootDirectory" type="folder">
    <FileTreeItem name="index.mdx" description="-> /" />
    <FileTreeItem name="getting-started.mdx" description="-> /getting-started" />
    <FileTreeItem name="guides" type="folder">
      <FileTreeItem name="index.mdx" description="-> /guides" />
      <FileTreeItem name="theming.mdx" description="-> /guides/theming" />
    </FileTreeItem>
  </FileTreeItem>
</FileTree>

- Routes are derived from file paths relative to `rootDirectory`.
- `.md`, `.mdx`, and `.openapi.json/.yaml/.yml` files are supported.
- The navigation structure can be customized through the `tabs` field in the configuration file; `pages` inside each tab defines the sidebar.

### HTML template

Each generated HTML file is based on the `index.html` template produced by the client build, with SSR content injected:

```html
<!doctype html>
<html lang="zh-CN">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>{siteTitle}</title>
    <!-- Vite injects built JS/CSS assets -->
  </head>
  <body>
    <div id="root">
      <!-- SSR: renderToString output -->
    </div>
  </body>
</html>
```

Clarify adjusts `<html lang>` / `dir` according to the current route language and injects the site description when the description meta tag is missing. Script and style assets are managed by Vite's HTML build process.

## Detailed build flow

### Step 1: client bundle build

Vite runs a normal build:
- Entry: `virtual:clarify/entry-client`
- Output: `output/assets/index-{hash}.js`, `output/assets/index-{hash}.css`, etc. (or the directory specified by `--output`)
- The client entry imports `virtual:clarify/routes`, `virtual:clarify/config`, and the OpenAPI registry, then hydrates the app.

### Step 2: route manifest generation

During plugin config resolution and build start:

1. Scan `rootDirectory` for `**/*.md`, `**/*.mdx`, and `**/*.openapi.*`.
2. For MDX / Markdown files, extract the frontmatter title and generate page sections from headings.
3. For OpenAPI files, parse the specification and generate page sections from the endpoint list.
4. Build the route manifest:

   ```typescript
   type ContentRoute = {
     path: string
     basePath?: string
     locale?: string
     title: string
     filePath: string
     virtualModuleId: string
     kind: 'mdx' | 'openapi'
     sections?: ContentSection[]
     contentArtifactUrl?: string
   }
   ```

5. Register virtual modules so `virtual:clarify/routes` can provide the route manifest and navigation tree.

### Step 3: SSR bundle build

After the client bundle build completes, `clarifyPlugin.closeBundle()`:

1. Creates a temporary `entry-server.ts`.
2. Builds it to `output/.ssr/entry-server.js` in Vite SSR mode.
3. Imports the SSR bundle.
4. Calls `render(url)` for each `ContentRoute`.

SSR uses `virtual:clarify/routes/server`, where page components are eager imports so the build phase can render synchronously.

### Step 4: HTML writing

Each route generates an independent HTML file from the `index.html` template in the client build output:

```txt
/                 → output/index.html
/getting-started  → output/getting-started/index.html
/en-US/guides     → output/en-US/guides/index.html
/404              → output/404/index.html and output/404.html
```

`404.mdx` remains a normal content route previewable at `/404`; SSG also writes an additional root-level `404.html` to match static hosting platform conventions for 404 filenames. Automatic file-tree navigation hides `404`, and it appears only when explicitly linked.

During writing, Clarify also updates:

- `<title>`;
- `description` meta;
- `keywords` meta;
- `<html lang>`;
- optional `<html dir>`.

### Step 5: content artifacts and `llms.txt`

The built-in content artifacts plugin attaches `contentArtifactUrl` to routes during `routes:resolved`, serves the corresponding content in the development server, and writes files during `build:done`.

These artifacts are used to:

- Copy raw Markdown / OpenAPI content.
- Copy raw content links.
- Provide readable documentation entry points for AI/LLM tools.
- Preserve original OpenAPI specifications.
- Generate `llms.txt`.

## Error behavior

If SSR fails for any route, the production build fails to avoid publishing incomplete documentation. This behavior is not configurable.

If best-effort output is needed in the future, it should be enabled through explicit configuration and documented with its trade-offs; the default behavior should continue to favor discovering problems at build time.

## Extension suggestions

| Requirement | Recommended hook |
|-------------|------------------|
| Generate a search index | `build:done` |
| Attach additional metadata to routes | `routes:resolved` |
| Provide virtual data to the Renderer | `modules:before` |
| Inject HTML tags or rewrite the shell | `html:transform` |
| Provide development-time middleware | `dev:configureServer` |

Keep output deterministic whenever possible so local builds and deployment previews are reproducible.
