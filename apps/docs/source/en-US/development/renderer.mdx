# Renderer architecture

The Renderer is Clarify's client runtime. It consumes processed content generated by the internal `@clarify-labs/cli` engine (routes, MDX components, and API data), then renders it as an interactive documentation site.

---

## Rendering philosophy

- **Static-first + client hydration**: HTML is generated at build time, then hydrated in the browser into a React Router app.
- **Build-time static data**: The CLI generates the route manifest, navigation tree, configuration, and OpenAPI registry at build time. The renderer consumes these static artifacts at runtime.
- **Component-driven UI**: MDX elements map to `@clarify-labs/renderer` primitives, and the app shell handles navigation, search, theme switching, and content actions.
- **Zero runtime configuration parsing**: All configuration is resolved at build time and baked into virtual modules.

---

## Runtime architecture

### Client entry

The CLI generates `virtual:clarify-entry-client` and injects that entry into the HTML. The entry imports renderer styles and virtual module data, then calls `render`:

```ts
import '@clarify-labs/renderer/style.css'
import { render } from '@clarify-labs/renderer/client'
import { routes, navigation } from 'virtual:clarify-routes'
import { config } from 'virtual:clarify-config'
import { openApis } from 'virtual:clarify-openapi-registry'

render({ config, routes, navigation, openApis })
```

User projects do not need to maintain `source/main.tsx` or a custom Vite entry.

### Server entry

The static generation phase uses a temporary SSR entry to call `renderToHTML`:

```ts
import { renderToHTML } from '@clarify-labs/renderer/server'
import { routes, navigation } from 'virtual:clarify-routes/server'
import { config } from 'virtual:clarify-config'
import { openApis } from 'virtual:clarify-openapi-registry'

export function render(url: string) {
  return renderToHTML({ config, routes, navigation, openApis, url })
}
```

### App shell

`AppShell` is the renderer's top-level shell. It receives configuration, routes, and the navigation tree:

```tsx
<AppShell config={config} routes={routes} navigation={navigation} />
```

**Layout regions**:

| Region | Responsibility | Fixed? |
|--------|----------------|--------|
| `Header` | Search, top navigation, language switcher, theme switcher | Yes (top) |
| `Navigation` | Hierarchical sidebar from `navigation` | Yes (left on desktop, drawer on mobile) |
| `Main` | Rendered page content (MDX or API page) | Scrollable |
| `ContentActions` | Copy page content, copy raw Markdown / OpenAPI, and related actions | Yes (top of content) |

## Virtual module system

The renderer depends on **virtual modules** generated by plugins at build time. They are the only bridge between plugins and the renderer.

### `virtual:clarify-routes`

Route manifest and navigation tree.

```typescript
export const routes: Array<{
  path: string
  title: string
  component: React.ComponentType
  kind: 'mdx' | 'openapi'
  sections?: Array<{ id: string; title: string }>
  contentArtifactUrl?: string
}>

export const navigation: NavigationTree
```

### `virtual:clarify-openapi-registry`

Parsed OpenAPI specification registry.

```typescript
export const openApis: Record<string, OpenAPISpec>
```

### `virtual:clarify-config`

Resolved project configuration and build options.

```typescript
export const config: ResolvedProjectConfig & ResolvedBuildOptions
```

### `virtual:clarify-page/*`

Each MDX / Markdown page is an independent virtual module:

```typescript
// virtual:clarify-page/getting-started
export { default } from '/absolute/path/to/source/getting-started.mdx'
```

## Route data consumed by the Renderer

The Renderer does not care about the file system; it only consumes route items generated by the CLI:

| Field | Purpose |
|-------|---------|
| `path` | React Router matching and navigation active state. |
| `title` | Page title and navigation display. |
| `description` / `keywords` | Runtime metadata updates after hydration. |
| `kind` | Distinguishes MDX and OpenAPI pages. |
| `locale`, `basePath`, `alternates`, `isFallback` | Language context and language switching. |
| `sections` | Sidebar anchors and in-page navigation. |
| `contentArtifactUrl` | Raw Markdown/OpenAPI content copy and link actions. |
| `component` / `lazy` | Page component import strategy. |

If a new route field needs to be consumed by the renderer, first confirm that the CLI can generate it stably for both client and server virtual modules.

## Loading, 404, and render error states

`PageSkeleton` is the Suspense fallback for lazy-loaded page modules. It simulates a real documentation structure, including title, summary, cards, body text, code blocks, and reference grids, so route transitions stay stable instead of showing blank placeholders.

Unknown client-side paths are handled by the catch-all route in `AppShell`:

- If a `/404` content page exists for the current language, Clarify reuses that MDX page as the not-found page.
- If there is no custom `/404`, Clarify displays the built-in localized 404 panel.
- Because unknown paths do not have an active content route, content actions and previous/next navigation stay hidden.

Page components are also wrapped with `PageErrorBoundary`. When a page throws during client rendering, Clarify displays a localized error panel with the current path, a refresh button, and expandable error details. Build-time SSR errors still follow the SSG `failOnError` strategy.

## Theme and UI semantic layers

The Renderer's visual system has two layers:

- **Theme token layer**: Resolved from project configuration and built-in themes. It expresses foundational design values such as brand colors, surface colors, borders, radii, and layout widths.
- **UI semantic layer**: Derived by the renderer from theme tokens. It expresses component roles such as menus, navigation, search, button states, and explanatory text.

Component authoring follows these principles:

- **Base properties can use theme tokens directly**: Stable foundational values such as backgrounds, borders, radii, brand colors, and content foreground colors can consume theme tokens directly.
- **Component roles should use the UI semantic layer**: Cross-component rules such as menu items, navigation links, search results, explanatory text, hover / active states, and menu widths should be expressed through semantic classes or variables.
- **Avoid repeating design rules inside components**: Do not scatter color mixing, opacity, font size, line height, or width rules across multiple components; centralize them in the renderer style layer.
- **Prefer adaptive layouts**: Use fluid grids, `minmax(0, 1fr)`, `w-full`, `h-full`, and explicit overflow boundaries. Do not write fixed preview widths/heights or one-off sizes in JSX.
- **Do not use Tailwind arbitrary values directly**: Font sizes, line heights, masks, grid tracks, widths, and heights that need to exceed the default scale should move into renderer CSS classes so component structure stays clear.
- **Do not over-variableize**: Only public theme capabilities, user-configurable options, or semantic rules reused across multiple components should be promoted to CSS variables. Private implementation values with no reuse plan and no intent to expose as configuration should remain ordinary CSS declarations in internal classes.
- **Previews must also follow the system**: Renderer entries related to preview / marketing can crop overflowing content, but private measurement values should stay in renderer CSS instead of being scattered across component JSX.
- **Minimum readable areas should be expressed through layout**: When components need readable width, express it through responsive layout, internal classes, and cropping behavior rather than fixed `w-*` snapshots.
- **Keep the theme system single-sourced**: The public theme is responsible only for foundational design capabilities; the internal semantic layer translates base tokens into concrete UI roles so Header, Navigation, Search, mobile drawers, and Portal menus remain consistent across themes.
- **Extend public theme capabilities carefully**: If future work needs finer-grained visual control, first confirm whether it belongs to user-configurable theme capability before promoting it to the theme token layer; otherwise keep it as an internal renderer semantic rule.
