# Renderer Architecture

`packages/renderer` provides the React runtime used by Clarify sites. It receives precomputed data from the CLI and renders a static-first, hydrated documentation application.

---

## Responsibilities

- Render the application shell, header, tabs, sidebar navigation, page navigation, footer, and content actions.
- Render MDX content through Clarify MDX primitives.
- Render OpenAPI reference pages and endpoint sections.
- Provide client and server entry points for hydration and SSG.
- Apply theme tokens and locale attributes at runtime.

The renderer should not load config files, scan the file system, or decide which content files exist. Those are CLI responsibilities.

---

## Runtime entries

The CLI-generated client module imports `@clarify-labs/renderer/client`:

```ts
import '@clarify-labs/renderer/style.css'
import { render } from '@clarify-labs/renderer/client'
import { routes, navigation } from 'virtual:clarify-routes'
import { config } from 'virtual:clarify-config'
import { openApis } from 'virtual:clarify-openapi-registry'

render({ config, routes, navigation, openApis })
```

The SSG bundle imports `@clarify-labs/renderer/server`:

```ts
import { renderToHTML } from '@clarify-labs/renderer/server'
import { routes, navigation } from 'virtual:clarify-routes/server'
import { config } from 'virtual:clarify-config'
import { openApis } from 'virtual:clarify-openapi-registry'

export function render(url: string) {
  return renderToHTML({ config, routes, navigation, openApis, url })
}
```

This keeps client hydration and server rendering aligned because both receive the same config, routes, navigation, and OpenAPI registry.

## AppShell

`AppShell` is the top-level UI boundary:

```tsx
<AppShell config={config} routes={routes} navigation={navigation} />
```

It is responsible for:

| Concern | Implementation detail |
|---------|-----------------------|
| Current route | Match `location.pathname` against `RouteItem.path`. |
| Current locale | Prefer `route.locale`, otherwise infer from the first URL segment and `config.i18n`. |
| Navigation | Select the correct localized navigation and active tab. |
| Sections | Pass current route H2/H3 sections into `SectionProvider`. |
| Page metadata | Update document title, description, and keywords on client navigation. |
| HTML locale | Set `<html lang>` and optional `dir`. |
| Theme | Convert resolved theme tokens into CSS variables. |
| Routing | Render page components through React Router. |

---

## Route data consumed by the renderer

The renderer consumes route items generated by the CLI. Important fields include:

| Field | Used for |
|-------|----------|
| `path` | React Router route matching and navigation active state. |
| `title` | Page title and navigation labels. |
| `description` / `keywords` | Runtime metadata updates after hydration. |
| `kind` | Distinguish MDX and OpenAPI pages when needed. |
| `locale`, `basePath`, `alternates`, `isFallback` | Locale switching and language context. |
| `sections` | Sidebar anchors and in-page navigation. |
| `contentArtifactUrl` | Raw Markdown/OpenAPI copy-link actions. |
| `component` / `lazy` | Page component import strategy. |

Do not add renderer-only route fields unless the CLI can generate them consistently for both client and server modules.

## MDX runtime

MDX pages are compiled by the CLI but rendered with renderer-provided components. This lets users write normal Markdown/MDX while Clarify controls typography, callouts, code blocks, cards, and other documentation primitives.

When adding a new MDX component:

1. implement it in `packages/renderer/source/components` or `mdx`;
2. export it through the renderer public surface if users should import it;
3. add it to the MDX component mapping when it should be available automatically;
4. document the user-facing API in feature docs.

---

## OpenAPI runtime

OpenAPI specs are parsed by the CLI and passed through `virtual:clarify-openapi-registry`. The renderer owns display concerns:

- operation grouping and endpoint cards;
- request examples and response examples;
- schema property rendering;
- error route display when parsing fails.

OpenAPI parsing, dereferencing, and diagnostics should stay in the CLI package.

---

## Theme and semantic UI layer

Resolved theme config provides stable tokens such as brand color, background, foreground, surface, muted color, border, code background, radius, and layout width. `AppShell` exposes them as CSS variables:

```txt
--clarify-theme-tokens-colors-primary
--clarify-theme-tokens-colors-background
--clarify-theme-tokens-radius-md
--clarify-theme-layout-max-width
```

Component styles should follow this rule:

- Use public theme tokens for stable brand and layout values.
- Use renderer-internal semantic classes or variables for component roles such as nav hover state, search result state, mobile drawer state, or active tab state.
- Avoid duplicating color mixing, spacing, and interaction rules across components.

---

## Design goals

- Keep renderer components reusable and tree-shakeable.
- Keep file-system and build logic out of the renderer.
- Prefer explicit TypeScript types over implicit runtime shapes.
- Keep SSR and client rendering deterministic.
- Make UI changes through shared components instead of route-specific hacks.
