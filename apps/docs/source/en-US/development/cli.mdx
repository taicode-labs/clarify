# CLI and engine design

The Clarify CLI lives in `packages/cli`. It wraps the internal build engine as `clarify dev`, `clarify build`, and `clarify init`. Users only need to care about Clarify commands and do not need to create or maintain Vite configuration files.

---

## Responsibility layers

| Layer | Responsibility | Audience |
|-------|----------------|----------|
| CLI commands | Argument parsing, development server, static build, project initialization | Clarify users |
| Internal engine | MDX/OpenAPI ingestion, routes, virtual modules, SSG | Clarify maintainers |
| Renderer | Page layout, navigation, code blocks, OpenAPI UI | Runtime components |

The CLI still reuses Vite's development server and build pipeline internally, but this is an implementation detail. The public API centers on `@clarify-labs/cli` and the `clarify` command.

---

## Content ingestion pipeline

The engine reads two types of source material:

| Source type | File pattern | Parser/tool | Output |
|-------------|--------------|-------------|--------|
| MDX/Markdown pages | `**/*.{md,mdx}` | `@mdx-js/rollup` + frontmatter parser | React component + metadata |
| OpenAPI specs | `*.openapi.yaml` / `*.openapi.json` | `swagger-parser` / `openapi-types` | Type-safe API reference data |

Pipeline overview:

```
source → Frontmatter → MDX/OpenAPI compilation → route manifest → Renderer → static HTML
```

1. **Frontmatter extraction**: Reads page metadata such as `title`, `description`, `keywords` / `keyword` / `tags`.
2. **OpenAPI asset loading**: Parses OpenAPI 3.0/3.1 specifications and injects them into the runtime registry.
3. **MDX compilation**: Compiles `.md` / `.mdx` into React components and injects Clarify's MDX component map.
4. **Route registration**: Maps file paths to URL paths, while generating sidebar navigation and H2/H3 headings.
5. **Content artifacts**: Attaches `contentArtifactUrl` to Markdown/OpenAPI routes for raw copy/link and `llms.txt`.
6. **Static generation**: Outputs an independent `index.html` for each route during the build phase.

## How CLI commands work

### `clarify dev`

1. Resolves the project root and content directory from CLI arguments.
2. Automatically creates a temporary HTML entry (if the project does not have `index.html`).
3. Starts the internal development server.
4. Watches content file changes and triggers route manifest rebuilds and page refreshes.

### `clarify build`

1. Uses the same content discovery logic to generate routes and virtual modules.
2. Bundles client assets.
3. Calls the server rendering entry to generate HTML for all routes.
4. Writes raw Markdown/OpenAPI files and `llms.txt`.

### `clarify init`

Generates a minimal project skeleton:

```
clarify.ts or clarify.json
source/index.mdx
package.json scripts
```

`clarify.ts` is the recommended format because it can use `defineConfig` for type hints and can host plugins.

## Routing system

Clarify uses convention-based routes derived from the file system:

```
source/
├── index.mdx           → /
├── getting-started.mdx → /getting-started
├── guides/
│   ├── index.mdx       → /guides
│   └── theming.mdx     → /guides/theming
└── api.openapi.yaml    → /api
```

Multilingual sites use locale directories. The default language has no prefix, and non-default languages use a locale prefix:

```
source/
├── zh-CN/index.md      → /
└── en-US/index.md      → /en-US
```

When `i18n.missing` is `fallback`, missing translations reuse the default-language page and mark the route with `isFallback`. Each localized route also carries `basePath` and `alternates` for language switching.

## Configuration loading

Configuration lookup order:

1. `clarify.ts`
2. `clarify.js`
3. `clarify.json`

`clarify.ts` is recommended for type-safe configuration:

```ts
import { defineConfig } from '@clarify-labs/cli'

export default defineConfig({
  title: 'My Docs',
  description: 'Product documentation',
  routePrefix: '/',
})
```

JSON configuration is validated and defaults are applied, but executable plugins cannot be expressed in JSON.

## Virtual modules

The internal engine passes build-time data to runtime through virtual modules:

- `virtual:clarify/config` — resolved site configuration and build options
- `virtual:clarify/routes` — client route manifest, lazy page imports, and navigation tree
- `virtual:clarify/routes/server` — server route manifest for SSR, with eager page imports
- `virtual:clarify/openapi` — collection of parsed OpenAPI specs
- `virtual:clarify/entry-client` — client entry injected by the CLI
- `virtual:clarify-page/*` — virtual module for each MDX/OpenAPI page

These modules are assembled internally by the CLI. Regular users do not need to import them directly.

---

## Hook lifecycle

Built-in plugins run first, and user plugins run afterward. The current lifecycle is:

```txt
routes:discover
  → routes:discovered
  → build default navigation
  → routes:resolved
  → buildVirtualModules
  → modules:before
  → html:transform
  → dev:configureServer / build closeBundle
  → build:done
```

When adding plugin capabilities, prefer modifying structured data (routes, navigation, modules) instead of rewriting generated artifacts directly. When a hook throws, the error includes the plugin name and hook name for easier diagnosis.

---

## Design principles

- **Hide build-tool details**: Users interact with the `clarify` command, not Vite configuration.
- **Content-first**: The default convention is `source`; adding a file adds a page.
- **Static-first**: Build artifacts are pure static files and can be deployed to any static hosting platform.
- **Extensible**: Internal hooks and the plugin system reserve extension points for future capabilities such as search and AI translation.
