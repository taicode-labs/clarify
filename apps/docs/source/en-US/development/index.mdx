# Developer Guide

This section explains how Clarify is designed internally. It is written for maintainers, contributors, and future developers who need to change the docs engine safely.

Clarify has one central product boundary: users write content and configuration, while Clarify owns the build pipeline, routing model, renderer runtime, and static output.

---

## Repository layout

```txt
apps/
  docs/       Documentation playground and integration site
  www/        Marketing site
packages/
  cli/        CLI, content engine, Vite integration, and SSG pipeline
  renderer/   React shell, MDX components, OpenAPI UI, and runtime entries
```

Use `source/` for source files. Do not add new `src/` directories in this repository.

---

## Mental model

Clarify is split into two major packages:

| Layer | Owns | Must not own |
|------|------|--------------|
| `packages/cli` | Config loading, content discovery, route and navigation models, plugin hooks, Vite orchestration, static generation, raw content artifacts | Visual presentation details that belong in React components |
| `packages/renderer` | Application shell, navigation UI, MDX primitives, OpenAPI rendering, hydration, client/server rendering entry points | File-system scanning, config file loading, or build orchestration |

The build-time data flow is:

```txt
clarify.ts/js/json
  → resolved project config
  → source/ content discovery
  → ContentRoute[] and NavigationTree
  → virtual modules
  → renderer client + server entries
  → output/ static HTML, assets, raw content, llms.txt
```

---

## Common commands

Run workspace commands from the repository root:

```bash
pnpm install
pnpm dev:docs
pnpm dev:www
pnpm typecheck
pnpm build
```

Package builds currently use Vite library builds. User-facing docs should still present `clarify dev` and `clarify build` as the public interface; Vite is an internal implementation detail.

## Design rules for future work

- Keep the CLI and renderer boundary explicit: the CLI prepares data; the renderer displays it.
- Prefer adding build-time metadata to `ContentRoute` and virtual modules instead of doing runtime file access.
- Keep generated output static and host-agnostic.
- Prefer `clarify.ts` for typed configuration and plugins; keep `clarify.js` and `clarify.json` supported for simpler projects.
- Document only implemented behavior, or clearly mark planned behavior as future work.
