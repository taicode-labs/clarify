# Development guide

This chapter is for developers who want to understand Clarify's implementation in depth, contribute to the project, or build extensions. The goal is not just to list files, but to explain Clarify's design boundaries: users write content and configuration, while Clarify is responsible for the content engine, routing model, rendering runtime, and static output.

---

## Repository structure

```txt
apps/
  docs/       Documentation playground and integration site
  www/        Marketing site
packages/
  cli/        CLI, content engine, Vite integration, and SSG pipeline
  renderer/   React app shell, MDX components, OpenAPI UI, and runtime entry points
```

The repository convention is to use `source/` as the source directory and not add `src/`.

---

## Core mental model

| Layer | Responsible for | Should not be responsible for |
|-------|-----------------|-------------------------------|
| `packages/cli` | Configuration loading, content discovery, routing/navigation model, plugin hooks, Vite orchestration, static generation, raw content artifacts | Specific visual presentation and React component layout |
| `packages/renderer` | App shell, navigation UI, MDX primitives, OpenAPI display, client hydration, server rendering entry | File system scanning, configuration file loading, build orchestration |

The build-time data flow is:

```txt
clarify.ts/js/json
  → resolved project configuration
  → source/ content discovery
  → ContentRoute[] and NavigationTree
  → virtual modules
  → renderer client/server entries
  → output/ static HTML, assets, raw content, llms.txt
```

---

## Quick start (developers)

```bash
git clone https://github.com/taicode-labs/clarify.git
cd clarify
pnpm install
pnpm dev:docs
pnpm dev:www
pnpm typecheck
pnpm build
```

Package builds currently use Vite library build. User-facing documentation should still treat `clarify dev` / `clarify build` as the public entry points; Vite is an internal CLI implementation detail.

## Future development principles

- Keep the CLI and Renderer boundaries clear: the CLI prepares data, and the Renderer displays data.
- Prefer putting build-time metadata into `ContentRoute` and virtual modules instead of accessing the file system at runtime.
- Keep the output static, self-hostable, and reproducible in `output/`.
- Prefer `clarify.ts` for type-safe configuration and plugins, while continuing to support `clarify.js` / `clarify.json`.
- Documentation should only describe implemented behavior; unimplemented content should be clearly marked as future plans.
