      <p className="text-sm/6 text-(--clarify-ui-text)">
        Click the button below to throw a client-side rendering error from this MDX page.
      </p>
      <div className="mt-4">
        <button
          type="button"
          className="rounded-(--clarify-theme-tokens-radius-md) bg-(--clarify-theme-tokens-colors-primary) px-3 py-2 text-sm/5 font-semibold text-white shadow-xs transition hover:opacity-90"
          onClick={() => setShouldThrow(true)}
        >
          Trigger render error
        </button>
      </div>
    </div>
  )

## 404 route behavior

Clarify treats `source/404.mdx` as a normal content route and also uses it as the fallback page for unknown client-side paths.

- The default `/404` route is output to both `output/404/index.html` and `output/404.html`.
- Language pages such as `/en-US/404` or `/zh-CN/404` can customize the not-found copy separately.
- Unknown client-side paths prefer the 404 page for the current language.
- Unknown paths do not have an active content route, so page content actions and previous/next navigation are hidden.

## Expected recovery panel

After the error is triggered, Clarify should display a localized recovery panel containing:

| Element | Expected behavior |
|---------|-------------------|
| Title and description | Explain that the current page failed to render. |
| Current path | Display the route that caused the failure. |
| Refresh action | Allow readers to retry by refreshing the page. |
| Details collapse | Keep stack information collapsed by default and expand it only when needed. |

```tsx
<PageErrorBoundary path={pathname}>
  <Suspense fallback={<PageSkeleton />}>
    <Routes>{/* page routes */}</Routes>
  </Suspense>
</PageErrorBoundary>
```
