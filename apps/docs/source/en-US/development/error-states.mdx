  const [shouldThrow, setShouldThrow] = useState(false)

  if (shouldThrow) {
    throw new Error('Intentional Clarify documentation render error preview')
  }

  return (
    <div className="not-prose my-6 rounded-xl border border-(--clarify-theme-tokens-colors-border) bg-(--clarify-theme-tokens-colors-surface) p-5 shadow-sm shadow-zinc-900/5">
      <p className="text-sm/6 text-(--clarify-ui-text)">
        Click the button below to throw a client-side render error from this MDX page.
      </p>
      <div className="mt-4">
        <button
          type="button"
          className="rounded-(--clarify-theme-tokens-radius-md) bg-(--clarify-theme-tokens-colors-primary) px-3 py-2 text-sm/5 font-semibold text-white shadow-xs transition hover:opacity-90"
          onClick={() => setShouldThrow(true)}
        >
          Trigger render error
        </button>
      </div>
    </div>
  )
}

## 404 route behavior

Clarify treats `source/404.mdx` as a normal content route and also uses it as the catch-all page for unknown client paths.

- The default `/404` route is emitted to both `output/404/index.html` and `output/404.html`.
- Locale-specific pages such as `/en-US/404` or `/zh-CN/404` can customize the not-found copy per locale.
- Unknown client routes reuse the active locale's 404 page when available.
- Page-specific content actions and previous/next links are hidden on unknown routes.

## Expected recovery panel

When the error is triggered, Clarify should show a localized panel with:

| Element | Expected behavior |
|---------|-------------------|
| Title and description | Explain that the current page failed to render. |
| Current path | Show the route that produced the failure. |
| Reload action | Allow the reader to retry by reloading the page. |
| Details disclosure | Keep stack traces collapsed until explicitly opened. |

```tsx
<PageErrorBoundary path={pathname}>
  <Suspense fallback={<PageSkeleton />}>
    <Routes>{/* page routes */}</Routes>
  </Suspense>
</PageErrorBoundary>
```
