# Getting Started

This page explains how Clarify works through the shortest possible path: install the CLI, create content, start the local site, and build static artifacts.

Clarify = MDX content + `clarify.ts` configuration + OpenAPI specs + static site generation. You mainly maintain `source/` and the configuration file, while Clarify handles the dev server, routing, rendering, and builds.

---

## 1. Prepare your environment

You need:

- Node.js ≥ 20
- pnpm ≥ 9

Create a new project with the official init command:

<CodeGroup>

```bash title="pnpm"
pnpm dlx @clarify-labs/cli init my-docs
cd my-docs
```

```bash title="npm"
npx @clarify-labs/cli init my-docs
cd my-docs
```

```bash title="yarn"
yarn dlx @clarify-labs/cli init my-docs
cd my-docs
```

</CodeGroup>

The scaffold adds scripts to `package.json` for you:

```json
{
  "scripts": {
    "dev": "clarify dev",
    "build": "clarify build --output output"
  }
}
```

---

## 2. Add site configuration

We recommend using `clarify.ts`, which gives you type hints and makes it easier to add plugins later.

```ts
import { defineConfig } from '@clarify-labs/cli'

export default defineConfig({
  title: 'My Docs',
  description: 'A documentation site built with Clarify',
  navbar: {
    links: [
      { label: 'Docs', href: '/getting-started' },
      { label: 'GitHub', href: 'https://github.com/example/project', external: true },
    ],
  },
})
```

Simple projects can also use `clarify.json`:

```json
{
  "title": "My Docs",
  "description": "A documentation site built with Clarify"
}
```

See the [Configuration Reference](/reference/clarify-config) for all fields.

---

## 3. Write your first pages

Clarify reads the `source/` directory by default. Adding an `.mdx` file adds a page.

```txt
my-docs/
├── clarify.ts
├── package.json
└── source/
    ├── index.mdx
    └── getting-started.mdx
```

`source/index.mdx`:

```mdx
---
title: Home
description: The home page for my product docs
---

# Welcome

This is the documentation home page.
```

`source/getting-started.mdx`:

```mdx
---
title: Getting Started
---

# Getting Started

This is the first guide.
```

Routes are generated automatically:

| File | Route |
|------|------|
| `source/index.mdx` | `/` |
| `source/getting-started.mdx` | `/getting-started` |
| `source/guides/index.mdx` | `/guides` |
| `source/guides/auth.mdx` | `/guides/auth` |

---

## 4. Start the dev server

Prefer starting through the project script so the same flow works whether the CLI is installed locally or available globally:

<CodeGroup>

```bash title="pnpm"
pnpm dev
```

```bash title="npm"
npm run dev
```

```bash title="yarn"
yarn dev
```

</CodeGroup>

Open the local address printed in the terminal. After you edit MDX, configuration, or OpenAPI files, the dev server regenerates the pages.

---

## 5. Build the production version

<CodeGroup>

```bash title="pnpm"
pnpm build
```

```bash title="npm"
npm run build
```

```bash title="yarn"
yarn build
```

</CodeGroup>

By default the output goes to `output/`:

```txt
output/
├── index.html
├── getting-started/
│   └── index.html
├── assets/
└── llms.txt
```

These are all static files, deployable to Vercel, Netlify, GitHub Pages, S3, Nginx, or any static hosting service.
