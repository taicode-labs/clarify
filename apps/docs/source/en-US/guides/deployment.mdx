# Publishing

This page brings together Clarify's publishing capabilities: build-time SSR, static generation, deployment to common platforms, subpath configuration, and outputting raw content and `llms.txt`.

Clarify supports SSR rendering, but it uses **build-time SSR**: `clarify build` renders each route into an HTML file during the build. The published artifacts are still pure static files that need no Node.js server runtime and can be deployed to any static hosting service.

---

## How SSR is supported

Clarify's production rendering flow is SSG (Static Site Generation) + hydration:

<Steps>
  <Step title="Start the production build">
    The build environment runs `pnpm exec clarify build`.
  </Step>
  <Step title="Render every route">
    The CLI uses a Vite SSR bundle to render each documentation route in Node.js.
  </Step>
  <Step title="Write static artifacts">
    Each route writes a standalone `index.html` and generates JS/CSS assets, raw Markdown/OpenAPI artifacts, and `llms.txt`.
  </Step>
  <Step title="Hydrate in the browser">
    Visitors first receive the rendered HTML. After JavaScript loads, React hydrates the page and subsequent in-site navigation runs on the client.
  </Step>
</Steps>

<Callout type="success" title="Static output, rich navigation">
  The production site needs only static file hosting. It still provides first-screen HTML, SEO-friendly deep links, and client-side navigation after hydration. A request-time Node.js SSR server is neither needed nor provided.
</Callout>

## Build artifacts

Run:

```bash
pnpm exec clarify build
```

By default it outputs to `output/`, but you can also specify a directory:

```bash
pnpm exec clarify build --output dist
```

The build result looks like:

```txt
output/
├── index.html
├── getting-started/
│   └── index.html
├── assets/
├── llms.txt
├── index.md
└── api.openapi.json
```

Every route has standalone HTML, suitable for SEO and direct access; after the browser loads the JavaScript it gets a client-side navigation experience. When deploying, only publish the `output/` directory — do not publish the temporary SSR bundle from the build process.

### Vercel

1. Push your code to GitHub/GitLab/Bitbucket.
2. Import the project in Vercel.
3. Build command: `pnpm exec clarify build`.
4. Output directory: `output`.

### Netlify

1. Connect the Git repository.
2. Build command: `pnpm exec clarify build`.
3. Publish directory: `output`.

### GitHub Pages

If the repository is named `my-repo` and the site address is `https://username.github.io/my-repo/`, configure:

```ts
export default defineConfig({
  routePrefix: '/my-repo/',
})
```

Example GitHub Actions:

```yaml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with:
          version: 9
      - run: pnpm install
      - run: pnpm exec clarify build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./output
```

### CDN / Nginx

Just upload `output/` to the web root:

```bash
rsync -avz output/ user@server:/var/www/html/
```

Nginx example:

```nginx
server {
    listen 80;
    root /var/www/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

## Raw content output

Clarify exposes raw content during both development and build, making it easy for search, knowledge bases, and AI tools to read.

| Page route | Raw content URL |
|----------|--------------|
| `/` | `/index.md` |
| `/getting-started` | `/getting-started.md` |
| `/features/openapi` | `/features/openapi.md` |
| `/api` | `/api.openapi.json` |

The content actions in the top-right corner of each page use these URLs to copy the raw content link, download the page's source content, or let AI tools reference a stable Markdown/OpenAPI address.

## `llms.txt`

The build artifacts include `llms.txt`, listing the site title, description, MDX pages, and OpenAPI documentation entries:

```txt
# Clarify

> An open-source documentation publishing tool built for MDX and OpenAPI.

## Docs
- [Getting Started](/getting-started.md)
- [API Documentation](/features/openapi.md)

## OpenAPI
- [API Reference](/api.openapi.json)
```

It is well suited to be:

- Read by AI coding assistants;
- Indexed by internal knowledge bases;
- Consumed by documentation quality-check tools;
- Used as an entry manifest by search or Q&A systems.

If `routePrefix` is configured, the links in `llms.txt` automatically include the same prefix.

## Pre-publish checklist

- The build environment meets Node.js `>=20.0.0` and pnpm `>=9.0.0`.
- `pnpm exec clarify build` completes successfully.
- `output/` contains the expected HTML, assets, raw Markdown/OpenAPI, and `llms.txt`.
- If deploying under a subpath, `routePrefix` is configured.
- The raw OpenAPI content does not contain internal information that should not be public.
- The deployment platform's output directory points to `output` or your custom output directory.
- The production environment is deployed as a static site — no Node.js SSR server is needed.
