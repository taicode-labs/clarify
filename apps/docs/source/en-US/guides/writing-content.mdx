# Writing Content

This is a writing specification that you can hand directly to a documentation author or an AI. A page completed with this guide should meet four conditions: a reader can finish one clear task, Clarify can build the page, site search can find its key sections, and an AI can understand and cite it through the raw Markdown and `llms.txt`.

<Callout type="tip" title="Shortest path">
  When you are unsure where to start, name the file after the user task, use `.md` by default, write one explicit H1 and a one-sentence goal, then divide prerequisites, actions, verification, and troubleshooting into H2 sections. Switch to `.mdx` only when the page needs Clarify components.
</Callout>

---

## From a requirement to a correct document

<Steps>
  <Step title="Define the reader and result">
    Complete the sentence ŌĆ£A [reader] needs to [result].ŌĆØ For example: ŌĆ£A backend developer creates an API key locally and verifies that the first request succeeds.ŌĆØ If the sentence contains two independent results, create two pages.
  </Step>
  <Step title="Choose the content type and location">
    Use `.md` for tutorials and conceptual explanations. Use `.mdx` when you need Tabs, Steps, Callouts, embedded OpenAPI, or custom React components. Put endpoint facts in `.openapi.json` or `.openapi.yaml`. The file path becomes the page route.
  </Step>
  <Step title="Write a completable main path first">
    Organize it as prerequisites, actions, and expected results. Commands and configuration examples must include enough context to complete the current step instead of presenting isolated fragments.
  </Step>
  <Step title="Choose the Clarify capability">
    Use `Steps` for ordered actions, `Tabs` for mutually exclusive alternatives, `Callout` for consequential information, `CodeGroup` for equivalent code samples, and OpenAPI for endpoint facts. Components should improve understanding, not decorate a page.
  </Step>
  <Step title="Preview, check, and build">
    In the development server, inspect the route, table of contents, links, and narrow-screen layout. Then run `clarify check` and `clarify build`. A successful build confirms that the MDX and OpenAPI sources meet the publishing contract.
  </Step>
</Steps>

### Ready-to-use page template

This template deliberately uses plain Markdown. Delete sections that do not apply. Rename the file to `.mdx` when you need components.

````md title="source/guides/create-api-key.md"
---
title: Create an API key
description: Create an API key and verify that it can access the Acme API.
keywords:
  - API key
  - authentication
---

# Create an API key

This guide helps backend developers create an API key and confirm it with a test request.

## Prerequisites

- You have administrator access to an Acme project.
- You have curl 8 or later installed.

## Create the key

1. Open the **API keys** page in the console.
2. Select **Create key**, enter a purpose, and save it.
3. Store the key in an environment variable immediately. It is shown only once.

```bash
export ACME_API_KEY="your-api-key"
```

## Verify the result

```bash
curl https://api.example.com/v1/me \
  --header "Authorization: Bearer $ACME_API_KEY"
```

A successful request returns the current account with HTTP status `200`.

## Troubleshooting

### The request returns 401

Check that the environment variable has no extra whitespace and that the key has not been revoked.

## Next step

Continue to [API Documentation](/features/openapi).
````

The template contains searchable headings, complete inputs, an observable result, and a next step. ŌĆ£Click the buttonŌĆØ or ŌĆ£run the command belowŌĆØ is not enough: name the control, explain the command's purpose, state the success signal, and describe how to recover from a likely failure.

## Choose a Clarify capability by content need

| What you are writing | What to use | Why |
|------|------|------|
| Installation, migration, or configuration flow | `Steps` / `Step` | Make sequence and completion boundaries explicit |
| Mutually exclusive language, framework, or platform options | `Tabs` / `Tab` | Let readers focus on the option that applies to them |
| Risk, prerequisite, or success result | `Callout` or `Note` | Separate consequential information from optional context |
| Equivalent implementations | `CodeGroup` | Compare and copy code in one place |
| Project structure | `FileTree` / `FileTreeItem` | Show exact file hierarchy and placement |
| Configuration fields or component props | `Properties` / `Property` | Present names, types, and descriptions consistently |
| Architecture, state, or call sequence | Mermaid code block | Keep diagrams versioned, searchable, and theme-aware |
| Feature entry points and recommended reading | `CardGroup` / `Card` | Create a clear path across pages |
| One endpoint inside a tutorial | `OpenApiOperation` | Keep endpoint facts in the spec and business context in the guide |
| A complete API reference | `.openapi.json/.yaml` or `OpenApiDocument` | Generate authentication, parameters, examples, and request code |
| Repeated product names, versions, and URLs | [Project Variables](/features/variables) | Maintain one value across MDX, frontmatter, and OpenAPI |
| Searchable and AI-readable output | Clear H2/H3 sections, raw content, and `llms.txt` | Clarify generates a full-text index and AI discovery entry point at build time |

See [Built-in Components](/reference/built-in-components) for complete props and examples, and [API Documentation](/features/openapi) for the OpenAPI workflow.

## Files are routes

Clarify generates page paths from the file structure under `source/`:

| File | Route |
|------|------|
| `source/index.md` | `/` |
| `source/getting-started.md` | `/getting-started` |
| `source/guides/index.md` | `/guides` |
| `source/guides/auth.mdx` | `/guides/auth` |
| `source/api.openapi.json` | `/api` |

Rules:

- `index.md` or `index.mdx` maps to the root path of its directory.
- File names should use lowercase and hyphens, e.g. `api-authentication.mdx`.
- `.md` and `.mdx` content files are supported, as well as `.openapi.json`, `.openapi.yaml`, and `.openapi.yml` spec files.
- If the site is deployed under a subpath, configure it with `routePrefix` ŌĆö see [Configuring the Site](/guides/navigation).

## Choose Markdown or MDX

Clarify parses `.md` and `.mdx` with different semantics. Both support frontmatter, <Tooltip content="GitHub Flavored Markdown">GFM</Tooltip>, syntax highlighting, Mermaid, heading navigation, and search. The difference is whether the page needs JSX capabilities.

| Capability | `.md` | `.mdx` |
|------|-------|--------|
| Standard Markdown and GFM | Yes | Yes |
| Raw HTML | Yes | Parsed as JSX syntax |
| Built-in or custom React components | No | Yes |
| `import` / `export` | No | Yes |
| JavaScript expressions `{...}` | No | Yes |

**Use `.md` by default.** It is a good fit for tutorials, guides, and reference content, and accepts regular HTML syntax:

```md
<img src="/images/overview.png">
```

**Choose `.mdx` only when the page needs components or expressions.** Tags in MDX follow JSX rules, so elements without children must be self-closing:

```mdx
<img src="/images/overview.png" />

<Callout type="tip">
  This content is rendered by a React component.
</Callout>

<img src="/clarify.png" alt="Clarify wordmark" caption="Images can include a caption and click-to-zoom preview directly." zoom />
```

Keeping regular content in `.md` prevents HTML from being interpreted as JSX while preserving strict diagnostics for real MDX component errors. Changing the extension does not change the route: both `source/guide.md` and `source/guide.mdx` map to `/guide`. Do not create both files for the same route.

## MDX basics

MDX files can contain plain Markdown and embedded JSX.

```mdx
# Heading

**bold**, *italic*, `inline code`.

- List item 1
- List item 2

> Blockquote
```

Embedding JSX:

```mdx
# Welcome

<div className="rounded-lg border p-4">
  This is a React snippet.
</div>
```

Importing custom components:

```mdx
import { Alert } from '../components/Alert'

# Page Title

<Alert type="warning">
  This is a warning message.
</Alert>
```

## Frontmatter

You can add YAML frontmatter at the top of each page for the page title and description:

```mdx
---
title: Page Title
description: The page description, used for SEO and social sharing
---

# Body content
```

Current stable fields:

| Field | Type | Description |
|------|------|------|
| `title` | `string` | The page title; when empty it tries to use the first `h1` |
| `description` | `string` | The page description, used for SEO meta |
| `keywords` / `keyword` / `tags` | `string \| string[]` | Page keywords, written into the HTML `keywords` meta |

> The current version does not provide advanced frontmatter fields such as `navOrder` or `hidden`. To control the sidebar, configure `tabs` explicitly in `clarify.ts`.

## Headings, the table of contents, and search

Clarify extracts H2/H3 from the content as page sections. Sections are used for in-page navigation and in-site search, so headings are not just typographic elements ŌĆö they are part of the information architecture.

```mdx
# API Documentation

## Authentication

### Create an access token

## Pagination
```

Suggestions:

- H1 expresses the page topic; if the frontmatter has no `title`, Clarify tries to use the H1 as the page title.
- H2 expresses a user task or a major question, e.g. "Create an access token" or "Deploy to GitHub Pages".
- H3 expresses sub-steps within a task or finer-grained scenarios.
- Avoid heavy use of weak headings like "Introduction", "More", or "Others".

Built-in search is now powered by Pagefind: production builds emit a `/pagefind/` index that can be deployed as static files, and the dev server also generates and serves a temporary index. Search results prefer full-text matches with highlighted excerpts; when the index is not available yet, Clarify falls back to quick jumps based on page titles, navigation nodes, paths, and H2/H3 section titles.

Multilingual sites load the index for the current language. Public Clarify locale codes use BCP 47, such as `zh-CN` and `en-US`, while the search index adapts internally to Pagefind's language-key behavior.

## Code blocks and GFM

Code blocks are highlighted automatically. Always label the language name to get more accurate syntax highlighting and a better copy experience:

```typescript
function greet(name: string): string {
  return `Hello, ${name}!`
}
```

If the language name cannot be recognized, Clarify falls back to rendering a plain code block without blocking the page build.

Built-in GitHub Flavored Markdown support:

| Syntax | Supported |
|------|----------|
| Tables | Yes |
| Task lists | Yes |
| Strikethrough | Yes |
| Autolinks | Yes |

```mdx
- [x] Tables supported
- [x] Task lists supported
- [ ] More writing experiences supported
```

## Mermaid diagrams

Use a fenced code block with the `mermaid` language to render diagrams. They are rendered on the client and automatically follow the light/dark theme:

````mdx
```mermaid
flowchart LR
  A[Write MDX] --> B{Has mermaid block?}
  B -- Yes --> C[Render diagram]
  B -- No --> D[Render code block]
```
````

The example above renders as:

```mermaid
flowchart LR
  A[Write MDX] --> B{Has mermaid block?}
  B -- Yes --> C[Render diagram]
  B -- No --> D[Render code block]
```

All Mermaid diagram types are supported (flowcharts, sequence diagrams, class diagrams, state diagrams, ER diagrams, Gantt charts, and more). If a diagram fails to parse, Clarify shows the error alongside the original source instead of breaking the page.

Large diagrams stay readable: each rendered diagram is zoomable and pannable. Scroll to zoom, drag to pan, double-click to reset, or use the zoom controls that appear on hover.

## Writing suggestions

- **One document solves one user task** ŌĆö don't create a separate document for each internal capability.
- **Lead with the goal and result, then move into the steps** instead of delaying the answer with product promotion or broad background.
- **Put field lists in the reference pages**, keeping only the most common configuration on task pages.
- **Use clear H2/H3 to carry search entries**, so users can jump from search results straight to a specific section.
- **Keep examples complete and consistent** so paths, variable names, ports, and return values do not contradict one another.
- **Let OpenAPI be the source of endpoint facts**, with tutorials only explaining the business flow.
- **Write content in a structure AI can also consume**: clear headings, explicit steps, and complete examples ŌĆö the built `.md` and `llms.txt` preserve this structure.

## Pre-publish checklist

Before submitting content, an author or AI should verify every item:

- [ ] The page solves one primary task and states the reader, goal, and completion result at the start.
- [ ] The file is in the correct content directory, uses a lowercase hyphenated name, and does not create a duplicate route.
- [ ] The page uses `.md` by default and uses `.mdx` only for JSX, built-in components, or imports.
- [ ] There is one H1, and each H2/H3 names a task or question instead of weak labels such as ŌĆ£IntroductionŌĆØ or ŌĆ£Other.ŌĆØ
- [ ] Every procedure includes prerequisites, actions, and an observable success signal.
- [ ] Code fences specify a language, examples include required context, and names and paths remain consistent throughout the page.
- [ ] Link text describes its destination instead of using isolated ŌĆ£click hereŌĆØ text or bare URLs.
- [ ] Callouts, Tabs, Steps, Cards, and OpenAPI components follow the selection rules above.
- [ ] The prose, examples, images, and OpenAPI files contain no secrets, internal addresses, or real user data.
- [ ] The local preview has a correct table of contents, search result, links, code-copy behavior, and mobile layout.
- [ ] `pnpm exec clarify check` and `pnpm exec clarify build` both finish successfully.

<Callout type="success" title="Definition of done">
  A reader who does not know the implementation can complete the task and determine whether it succeeded from this page alone. Search can jump directly to the key sections. An AI without extra context can restate the prerequisites, steps, examples, and result. The document is complete only when all three are true.
</Callout>
