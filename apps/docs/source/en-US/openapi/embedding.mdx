# OpenAPI embedding guide

Clarify can render `.openapi.json` / `.openapi.yaml` files as complete API References, and it also lets you embed a single endpoint or a complete Reference inside MDX guides. This lets you place “endpoint facts” and “business workflow explanations” on the same page.

---

## Use cases

| Use case | Recommended approach |
|----------|----------------------|
| Explain which endpoint to call for a step in a tutorial | `OpenApiOperation` |
| Show complete parameters and responses for a core endpoint on a topic page | `OpenApiOperation` |
| Add explanatory content in an MDX page before showing a complete API Reference | `OpenApiDocument` |
| Let readers see business explanations and machine-readable specs together | MDX explanation + OpenAPI embedding + raw spec link |

---

## Prepare an OpenAPI file

Place the spec under the current locale directory, for example:

<FileTree aria-label="OpenAPI spec under a locale directory">
  <FileTreeItem name="source" type="folder">
    <FileTreeItem name="zh-CN" type="folder">
      <FileTreeItem name="api.openapi.json" />
      <FileTreeItem name="openapi" type="folder">
        <FileTreeItem name="embedding.mdx" />
      </FileTreeItem>
    </FileTreeItem>
  </FileTreeItem>
</FileTree>

The file name determines the Reference route:

| File | Reference route | Raw spec output |
|------|-----------------|-----------------|
| `api.openapi.json` | `/api` | `/api.openapi.json` |
| `billing.openapi.yaml` | `/billing` | `/billing.openapi.yaml` |
| `admin.openapi.yml` | `/admin` | `/admin.openapi.yml` |

In multilingual sites, English specs can be placed at `source/en-US/api.openapi.json`, and Chinese specs at `source/zh-CN/api.openapi.json`. Clarify reads the corresponding file for the current language and then applies the i18n fallback strategy when it is missing.

## Embed a single endpoint

Use `OpenApiOperation`:

```mdx
<OpenApiOperation specPath="/api" path="/examples/basic-request" method="get" />
```

Parameters:

| Parameter | Required | Description |
|-----------|----------|-------------|
| `specPath` | Yes | Route of the OpenAPI Reference, without the `.openapi.json` suffix, such as `/api` |
| `path` | Yes | Path in OpenAPI `paths`; it must exactly match the spec, such as `/examples/path-parameters/{pageId}` |
| `method` | Yes | HTTP method; lowercase is recommended, such as `get`, `post`, `patch` |

Rendered result:

<OpenApiOperation specPath="/api" path="/examples/basic-request" method="get" />

The play button in the operation panel opens an interactive request tester. If sending the request is itself part of the guide flow, you can embed the same tester directly:

```mdx
<OpenApiRequest specPath="/api" path="/examples/basic-request" method="get" />
```

`OpenApiRequest` takes the same parameters as `OpenApiOperation`. It renders only the request workbench, which is ideal when a tutorial wants readers to actually send a request and inspect the response.

## Embed a complex request endpoint

When an endpoint contains multiple request bodies, multiple examples, multi-status responses, or callbacks, you can embed it with the same component. The following `POST /examples/complex-request-body` covers JSON / YAML request bodies, multiple examples, JSON / text responses, and error responses:

```mdx
<OpenApiOperation specPath="/api" path="/examples/complex-request-body" method="post" />
```

Rendered result:

<OpenApiOperation specPath="/api" path="/examples/complex-request-body" method="post" />

## Embed a complete Reference

If you want to write business context first and then display a complete API Reference in an MDX page, use `OpenApiDocument`:

```mdx
# API Reference

Below is the complete endpoint list for the current service.

<OpenApiDocument specPath="/api" />
```

This is useful for:

- Adding authentication, rate-limit, and error format notes before the Reference.
- Combining “explanation + complete endpoints” on a product page.
- Temporarily placing the complete Reference in a topic document.

## Common troubleshooting

### `OpenAPI spec not found`

Check:

- Whether the OpenAPI file is located under `source` or the current locale directory.
- Whether the file name is `.openapi.json` / `.openapi.yaml` / `.openapi.yml`.
- Whether `specPath` is a route such as `/api`, not a file name like `api.openapi.json`.

### `Endpoint not found`

Check:

- Whether `path` exactly matches the OpenAPI `paths` key.
- Whether path parameters use OpenAPI syntax such as `{pageId}`, not `:pageId`.
- Whether `method` exists and its casing matches the component convention.

### Page build failed

Production builds parse and dereference OpenAPI specifications. If the spec is invalid, the build fails. First confirm that the JSON / YAML syntax is correct, then check whether `$ref` can be resolved.

## Recommended practices

- Use OpenAPI specs to maintain the real endpoint structure; do not hand-write parameter tables in MDX.
- Use MDX to explain business workflows, authentication prerequisites, common errors, and best practices.
- Embed `OpenApiOperation` next to key steps to reduce back-and-forth between guides and the Reference.
- Use the same spec for the complete Reference, embedded endpoints, raw spec output, and `llms.txt`.
