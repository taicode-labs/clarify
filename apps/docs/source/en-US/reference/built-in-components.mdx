# Built-in public components

Clarify registers a set of public components in MDX by default. You do not need to `import` them manually; you can use them directly in `.mdx` files to organize homepage entries, hints, action buttons, and code examples.

---

## Automatically available MDX components

The following components can be used directly in MDX pages:

- Content hints: `Callout`, `Note`
- Entries and actions: `CardGroup` / `Card`, `Button`
- Code and collapsible content: `CodeGroup`, `Collapse`
- Page organization: `Steps` / `Step`, `Tabs` / `Tab`
- Layout and property descriptions: `Row` / `Col`, `Properties` / `Property`
- Content embedding: `WebFrame`
- Markdown rendering: `Markdown`
- OpenAPI: `OpenApiOperation`, `OpenApiDocument`

## Callout

`Callout` is suitable for notes, reminders, or risk warnings that are strongly related to the main text.

### Basic usage

```mdx
<Callout type="info">
  The model list is updated continuously. Before production integration, use the models currently available in the console or model marketplace as the source of truth.
</Callout>
```

### Types

`type` supports `info`, `note`, `tip`, `warning`, `danger`, and `success`.

<Callout type="note" title="Note">
  Best for adding contextual explanations without interrupting the main flow with details.
</Callout>

<Callout type="tip" title="Tip">
  Best for recommended practices, efficiency tips, or better paths.
</Callout>

<Callout type="warning" title="Warning">
  Best for caveats that may cause build failures, insufficient permissions, or behavior changes.
</Callout>

<Callout type="danger" title="Danger">
  Best for high-risk operations, such as deleting data, resetting environments, or exposing secrets.
</Callout>

<Callout type="success" title="Success">
  Best for showing that an operation is complete, configuration has taken effect, or verification has passed.
</Callout>

### Props

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `type` | `'info' \| 'note' \| 'tip' \| 'warning' \| 'danger' \| 'success'` | `'info'` | Controls the color and icon |
| `title` | `ReactNode` | Generated automatically from `type` | Title; pass an empty value to hide it |
| `icon` | `boolean` | `true` | Whether to show the icon |
| `className` | `string` | - | Additional CSS class names |

## Note

`Note` is a lighter inline hint component, suitable for supplementary information that does not change the task flow. Compared with `Callout`, it does not require a type and has no title bar.

```mdx
<Note>
  Titles, H2/H3 headings, and navigation names affect site search results. Use terms users can understand.
</Note>
```

<Note>
  Titles, H2/H3 headings, and navigation names affect site search results. Use terms users can understand.
</Note>

### Props

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `children` | `ReactNode` | Required | Hint content |

## CardGroup / Card

`CardGroup` and `Card` are suitable for documentation entries, feature lists, solution matrices, or section navigation.

### Basic usage

```mdx
<CardGroup cols={2}>
  <Card title="About Clarify" icon="badge-info" href="/getting-started">
    Learn Clarify's product positioning, core value, and recommended usage path.
  </Card>
  <Card title="Quick start" icon="rocket" href="/getting-started">
    Prepare the environment, install the CLI, and create your first documentation site.
  </Card>
</CardGroup>
```

Rendered result:

<CardGroup cols={2}>
  <Card title="About Clarify" icon="badge-info" href="/getting-started">
    Learn Clarify's product positioning, core value, and recommended usage path.
  </Card>
  <Card title="Quick start" icon="rocket" href="/getting-started">
    Prepare the environment, install the CLI, and create your first documentation site.
  </Card>
  <Card title="Writing docs" icon="file-code-2" href="/guides/writing-content">
    Use Markdown, JSX, and built-in components to write richer content.
  </Card>
  <Card title="API documentation" icon="braces" href="/features/openapi">
    Generate endpoint documentation and single-endpoint displays from OpenAPI description files.
  </Card>
</CardGroup>

### Icon naming

The `icon` prop of `Card` uses `lucide-react` icon names and supports automatic conversion from kebab-case. For example:

| Syntax | Corresponding icon |
|--------|--------------------|
| `badge-info` | `BadgeInfo` |
| `rocket` | `Rocket` |
| `file-code-2` | `FileCode2` |
| `shield-check` | `ShieldCheck` |

### Props

#### CardGroup

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `cols` | `1 \| 2 \| 3 \| 4` | `2` | Number of desktop columns |
| `className` | `string` | - | Additional CSS class names |

#### Card

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `title` | `string` | Required | Card title |
| `icon` | `string` | - | lucide icon name; kebab-case is supported |
| `href` | `string` | - | Card link; internal paths use client-side routing |
| `className` | `string` | - | Additional CSS class names |

## Collapse

`Collapse` is suitable for collapsing optional supplementary notes, implementation details, or long lists to keep the main documentation flow concise.

```mdx
<Collapse title="View details">
  This content is collapsed by default and is displayed only after the user clicks the title.
</Collapse>

<Collapse summary="Open by default" defaultOpen>
  This content is initially visible, and users can still collapse it manually.
</Collapse>
```

<Collapse title="View details">
  This content is collapsed by default and is displayed only after the user clicks the title.
</Collapse>

### Props

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `title` | `ReactNode` | - | Collapse panel title; takes precedence over `summary` |
| `summary` | `ReactNode` | `'Details'` | Fallback title |
| `defaultOpen` | `boolean` | `false` | Whether to expand by default |
| `className` | `string` | - | Additional CSS class names |

## Button

`Button` is suitable for providing a clear next action in documentation.

```mdx
<Button href="/getting-started" arrow="right">
  Start installation
</Button>

<Button href="https://github.com/taicode-labs/clarify" variant="outline">
  View GitHub
</Button>
```

Rendered result:

<div className="not-prose my-6 flex flex-wrap gap-3">
  <Button href="/getting-started" arrow="right">
    Start installation
  </Button>
  <Button href="https://github.com/taicode-labs/clarify" variant="outline">
    View GitHub
  </Button>
</div>

### Props

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `variant` | `'primary' \| 'secondary' \| 'filled' \| 'outline' \| 'text'` | `'primary'` | Button style |
| `arrow` | `'left' \| 'right'` | - | Whether to show a directional arrow |
| `href` | `string` | - | Renders as a link when present; otherwise renders as a button |

## Steps / Step

`Steps` and `Step` are used to guide users through sequential tasks with numbered steps. They are ideal for installation guides, setup instructions, tutorials, and any multi-step process.

### Basic usage

```mdx
<Steps>
  <Step title="Install dependencies">
    Run the following command to install Clarify CLI:
    `pnpm add -D @clarify-labs/cli`
  </Step>

  <Step title="Create configuration file">
    Create a `clarify.ts` file in your project root with the configuration.
  </Step>

  <Step title="Start development server">
    Run the dev server to preview your documentation:
    `pnpm exec clarify dev`
  </Step>
</Steps>
```

Rendered result:

<Steps>
  <Step title="Install dependencies">
    Run the following command to install Clarify CLI: `pnpm add -D @clarify-labs/cli`
  </Step>
  <Step title="Create configuration file">
    Create a `clarify.ts` file in your project root with your documentation title and navigation structure.
  </Step>
  <Step title="Start development server">
    Run the dev server to preview your documentation: `pnpm exec clarify dev`
  </Step>
</Steps>

### With custom icons

You can use Lucide icon names (kebab-case or PascalCase) for step indicators:

```mdx
<Steps>
  <Step title="Download" icon="download">
    Download the installation package.
  </Step>
  <Step title="Install" icon="package">
    Run the installer and follow the prompts.
  </Step>
  <Step title="Configure" icon="settings">
    Set up your preferences and API keys.
  </Step>
</Steps>
```

<Steps>
  <Step title="Download" icon="download">
    Download the installation package.
  </Step>
  <Step title="Install" icon="package">
    Run the installer and follow the prompts.
  </Step>
  <Step title="Configure" icon="settings">
    Set up your preferences and API keys.
  </Step>
</Steps>

### Custom numbering

You can override the automatic numbering with the `stepNumber` prop:

```mdx
<Steps>
  <Step title="Prerequisites" stepNumber={0}>
    Check system requirements before installation.
  </Step>
  <Step title="Install">
    Install the package.
  </Step>
  <Step title="Verify">
    Verify the installation was successful.
  </Step>
</Steps>
```

### Props

#### Steps

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `titleSize` | `'p' \| 'h2' \| 'h3' \| 'h4'` | `'h3'` | HTML tag for step titles |
| `className` | `string` | - | Additional CSS class names |

#### Step

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `title` | `string` | Required | Step title |
| `icon` | `string \| ReactNode` | - | Lucide icon name or custom React element |
| `stepNumber` | `number` | Auto-numbered | Override automatic numbering |
| `id` | `string` | Auto-generated | HTML id for anchor linking; auto-generated from title if not provided |
| `noAnchor` | `boolean` | `false` | Hide the anchor link icon |
| `titleSize` | `'p' \| 'h2' \| 'h3' \| 'h4'` | `'h3'` | HTML tag for this step's title; inherits from `Steps` if not set |
| `className` | `string` | - | Additional CSS class names |

## Tabs / Tab

`Tabs` and `Tab` are used to organize related content in switchable panels. They are ideal for showing code examples in different languages, configuration for different platforms, or alternative approaches to the same task.

### Basic usage

```mdx
<Tabs>
  <Tab title="npm">
    Install: `npm install @clarify-labs/cli`
  </Tab>
  <Tab title="pnpm">
    Install: `pnpm add @clarify-labs/cli`
  </Tab>
  <Tab title="yarn">
    Install: `yarn add @clarify-labs/cli`
  </Tab>
</Tabs>
```

Rendered result:

<Tabs>
  <Tab title="npm">
    Install using npm: `npm install @clarify-labs/cli`
  </Tab>
  <Tab title="pnpm">
    Install using pnpm: `pnpm add @clarify-labs/cli`
  </Tab>
  <Tab title="yarn">
    Install using yarn: `yarn add @clarify-labs/cli`
  </Tab>
</Tabs>

### With icons

Add visual indicators to tabs using Lucide icon names:

```mdx
<Tabs>
  <Tab title="JavaScript" icon="javascript">
    Example: `export default function hello() { console.log('Hello, world!') }`
  </Tab>
  <Tab title="TypeScript" icon="braces">
    Example: `export default function hello(): void { console.log('Hello, world!') }`
  </Tab>
  <Tab title="Python" icon="code">
    Example: `def hello(): print('Hello, world!')`
  </Tab>
</Tabs>
```

### Synchronized tabs

By default, tabs with the same title are synchronized across the page. When a user selects a tab, all other tab groups with matching titles will update automatically. This is useful for maintaining consistent language or platform selection throughout documentation.

```mdx
<Tabs>
  <Tab title="macOS">
    Instructions for macOS users.
  </Tab>
  <Tab title="Windows">
    Instructions for Windows users.
  </Tab>
  <Tab title="Linux">
    Instructions for Linux users.
  </Tab>
</Tabs>

Later in the same page:

<Tabs>
  <Tab title="macOS">
    Additional macOS-specific information.
  </Tab>
  <Tab title="Windows">
    Additional Windows-specific information.
  </Tab>
  <Tab title="Linux">
    Additional Linux-specific information.
  </Tab>
</Tabs>
```

<Callout type="tip">
  When a user clicks on "Windows" in the first tab group, the second tab group will automatically switch to "Windows" as well. To disable this behavior, set `sync={false}`.
</Callout>

### Props

#### Tabs

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `defaultTabIndex` | `number` | `0` | Index of the initially selected tab |
| `sync` | `boolean` | `true` | Whether to synchronize selection across all tab groups with matching titles |
| `borderBottom` | `boolean` | `true` | Add a bottom border below the tab list |
| `className` | `string` | - | Additional CSS class names |

#### Tab

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `title` | `string` | Required | Tab label |
| `icon` | `string \| ReactNode` | - | Lucide icon name or custom React element |
| `id` | `string` | - | HTML id for the tab panel |
| `className` | `string` | - | Additional CSS class names |

## WebFrame

`WebFrame` is used to embed web content in documentation pages, such as product screens, hosted demos, generated previews, or other internal pages. It renders a simulated browser window by default, and you can disable that chrome with `showWindow={false}`.

### Basic usage

```mdx
<WebFrame src="https://clarify.pub" title="Theme preview" />
```

Rendered result:

<WebFrame src="https://clarify.pub" title="Theme preview" height={360} />

### Without simulated window

```mdx
<WebFrame src="https://clarify.pub" title="Components preview" showWindow={false} height={320} />
```

### Absolute URL

Use a full URL with protocol when embedding an external page.

```mdx
<WebFrame src="https://clarify.pub" title="Clarify website" height={320} />
```

Rendered result:

<WebFrame src="https://clarify.pub" title="Clarify website" height={320} />

### Inline content

If you do not pass `src`, `WebFrame` renders its children inside the frame. This is useful for small product UI mockups or static previews.

```mdx
<WebFrame title="Inline preview" height={180}>
  <div style={{ padding: 24 }}>
    <strong>Preview content</strong>
    <p>This content is rendered directly inside the frame.</p>
  </div>
</WebFrame>
```

### Props

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `src` | `string` | - | URL to embed in an iframe |
| `title` | `string` | `'Embedded content'` | Accessible iframe title and simulated window label |
| `showWindow` | `boolean` | `true` | Whether to render the simulated browser window chrome |
| `height` | `CSSProperties['height']` | `420` | Embedded content height |
| `className` | `string` | - | Additional CSS class names for the outer container |
| `iframeClassName` | `string` | - | Additional CSS class names for the iframe or inline content container |
| `children` | `ReactNode` | - | Inline content rendered when `src` is not provided |

## Row / Col

`Row` and `Col` are used to create two-column layouts on large screens. They are common in documentation structures like “explanation on the left, example on the right” or “steps on the left, endpoint on the right.”

````mdx
<Row>
  <Col>
    ## Writing advice

    First explain why the user should perform this step.
  </Col>
  <Col sticky>
    ```bash
    pnpm exec clarify dev
    ```
  </Col>
</Row>
````

### Props

#### Row

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `children` | `ReactNode` | Required | Two-column content |

#### Col

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `children` | `ReactNode` | Required | Content of the current column |
| `sticky` | `boolean` | `false` | Whether to stick to the top on large screens |

## Properties / Property

`Properties` and `Property` are suitable for describing configuration options, component props, or API fields. They are better than regular tables for multi-line descriptions.

```mdx
<Properties>
  <Property name="routePrefix" type="string">
    Deployment subpath, such as `/docs/`.
  </Property>
  <Property name="tabs" type="ClarifyTabsConfig">
    Configure top-level business categories and sidebar structures.
  </Property>
</Properties>
```

### Props

#### Properties

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `children` | `ReactNode` | Required | One or more `Property` components |

#### Property

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `name` | `string` | Required | Property name |
| `type` | `string` | - | Type description |
| `children` | `ReactNode` | Required | Property description |

## OpenAPI Components

`OpenApiOperation` is used to embed a single endpoint in a guide, and `OpenApiDocument` is used to embed a complete API Reference.

```mdx
<OpenApiOperation specPath="/api" path="/sessions" method="post" />

<OpenApiDocument specPath="/api" />
```

| Component | Key props | Description |
|-----------|-----------|-------------|
| `OpenApiOperation` | `specPath`, `path`, `method` | Displays a single endpoint, suitable for task-oriented guides |
| `OpenApiDocument` | `specPath` | Displays a complete OpenAPI Reference |

The OpenAPI panel displays servers, authentication, parameters, request bodies, response examples, and multi-language request code based on the specification. For more details, see [API documentation](/features/openapi).

## Usage suggestions

- Prefer `CardGroup` / `Card` for homepages, section indexes, and feature matrices.
- Use `Callout` for limitations, risks, and supplementary notes strongly related to the main text; use `Note` for lightweight supplementary notes.
- Use `Button` when guiding users to continue reading or perform an action.
- Use `Steps` / `Step` for sequential tasks, installation guides, tutorials, and multi-step processes.
- Use `Tabs` / `Tab` for code examples in different languages, platform-specific instructions, or alternative approaches; leverage `sync={true}` to maintain consistent selection across the page.
- Use `CodeGroup` for examples covering multiple languages, environments, or package managers.
- Use `Collapse` for optional details, long lists, and advanced content.
- Use `WebFrame` for embedded demos, previews, and product screens; disable `showWindow` when the embedded content already has its own chrome.
- Prefer `Properties` / `Property` for configuration options, component props, and field descriptions.
- Explain processes in guides, and leave endpoint facts to `OpenApiOperation` or `OpenApiDocument`.

## Custom MDX components

If you need project-specific components, import them explicitly at the top of the MDX file:

```mdx
import { Alert } from '../components/Alert'

<Alert type="warning">
  This is an example of a custom React component.
</Alert>
```

Clarify's built-in components are registered automatically in MDX; custom components need to be imported manually.
