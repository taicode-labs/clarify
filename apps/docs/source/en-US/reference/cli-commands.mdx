# CLI Commands Reference

`@clarify-labs/cli` provides the `clarify` command, which wraps the dev server, static build, and project initialization. Users do not need to create a Vite configuration file.

---

## Installation

For a new project, the shortest path is to scaffold it directly with the published package:

<CodeGroup>

```bash title="pnpm"
pnpm dlx @clarify-labs/cli init my-docs
cd my-docs
pnpm dev
```

```bash title="npm"
npx @clarify-labs/cli init my-docs
cd my-docs
npm run dev
```

```bash title="yarn"
yarn dlx @clarify-labs/cli init my-docs
cd my-docs
yarn dev
```

</CodeGroup>

If you want to add Clarify to an existing project, install the CLI as a project dev dependency so the CLI version is locked and reviewed alongside the repository:

<CodeGroup>

```bash title="pnpm"
pnpm add -D @clarify-labs/cli
```

```bash title="npm"
npm install -D @clarify-labs/cli
```

```bash title="yarn"
yarn add -D @clarify-labs/cli
```

</CodeGroup>

Then you can run the CLI directly via `pnpm exec`:

```bash
pnpm exec clarify dev
pnpm exec clarify check
pnpm exec clarify build
```

Or add scripts to `package.json` and prefer running through the package manager script entry:

```json title="package.json"
{
  "scripts": {
    "dev": "clarify dev",
    "build": "clarify build"
  }
}
```

For a quick trial, you can also install it globally and run `clarify dev` / `clarify build` directly, but for team projects we recommend preferring a project-local dependency.

## `clarify init`

Creates a Clarify project skeleton and automatically updates the `dev` and `build` scripts in `package.json`.

```bash
npx @clarify-labs/cli init my-docs
```

By default, `clarify init` generates or updates the following:

- `clarify.ts`
- `source/index.mdx`
- The `dev` / `build` scripts in `package.json`
- The `@clarify-labs/cli` dev dependency

If you want to install dependencies immediately after initialization, use:

```bash
npx @clarify-labs/cli init my-docs --install
```

When `--install` is not used, the command prints the next steps, for example:

```bash
cd <project-root>
npm install
npm run dev
```

Common options:

| Option | Description | Default |
|------|------|--------|
| `[directory]` | The target project directory | Current directory |
| `--content <dir>` | The content directory | `source` |
| `--template <name>` | The `minimal`, `standard`, or `complete` template | `standard` |
| `--force` | Overwrite existing initialization files | `false` |
| `--install` | Install dependencies after initialization | `false` |

Template use cases:

| Template | Best for |
|------|----------|
| `minimal` | Quickly verifying Clarify or connecting an existing content directory |
| `standard` | Creating a regular documentation site with the recommended base structure |
| `complete` | Learning the full capabilities by examining navigation, components, OpenAPI, and publishing configuration examples |

## `clarify dev`

Starts the local documentation dev server.

```bash
npm run dev
```

This is the preferred user-facing entry because it works the same way whether `clarify` comes from a local dependency or a global install. The underlying script typically runs `clarify dev`.

<CodeGroup>

```bash title="pnpm"
pnpm dev
```

```bash title="npm"
npm run dev
```

```bash title="yarn"
yarn dev
```

</CodeGroup>

Common options:

| Option | Description | Default |
|------|------|--------|
| `--root <dir>` | The project root directory | Current directory |
| `--content <dir>` | The content directory | `source` |
| `--host [host]` | The listen address | Vite default |
| `--port <port>` | The port number | Vite default |
| `--open [path]` | Open the browser after startup | `false` |

Example:

```bash
pnpm exec clarify dev --port 5173 --open
```

---

## `clarify build`

Builds a deployable static site.

```bash
pnpm exec clarify build
```

Common options:

| Option | Description | Default |
|------|------|--------|
| `--root <dir>` | The project root directory | Current directory |
| `--content <dir>` | The content directory | `source` |
| `--output <dir>` | The output directory | `output` |

Example:

```bash
pnpm exec clarify build --output dist
```

When `siteUrl` is configured, the build also outputs `sitemap.xml` and `robots.txt`.

---

## `clarify check`

Checks the documentation project without producing build files.

```bash
pnpm exec clarify check
```

`check` validates the content directory, detects duplicate routes, flags i18n fallback routes, and verifies local Markdown links. It is useful in CI or before publishing a docs change.

Common options:

| Option | Description | Default |
|------|------|--------|
| `--root <dir>` | The project root directory | Current directory |
| `--content <dir>` | The content directory | `source` |
| `--strict` | Treat warnings as failures too | `false` |
| `--format <format>` | `text` or `json` | `text` |

## Configuration file

We recommend using `clarify.ts` as the single project configuration entry. It can configure both the site display fields and the build-time extension capabilities:

```ts
import { defineConfig } from '@clarify-labs/cli'

export default defineConfig({
  title: 'Clarify Docs',
  description: 'Documentation powered by Clarify',
  theme: { preset: 'default' },
  navigation: {
    tabs: [
      {
        tab: 'Docs',
        icon: 'BookOpen',
        pages: 'FileTree',
      },
    ],
  },
  plugins: [
    // searchPlugin(),
    // translatePlugin(),
  ],
})
```

Clarify looks for the configuration file in the following priority order:

```
clarify.ts > clarify.js > clarify.json
```

`clarify.json` is good for simple projects that do not need plugin functions; use `clarify.ts` when you need plugins or stronger type hints.

---

## Configuration priority

CLI options control runtime paths and the output directory, while the configuration file controls site display and build-time extensions. The priority is:

```
CLI options > clarify.ts/js/json > defaults
```
