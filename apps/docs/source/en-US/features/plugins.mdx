# Plugins

Clarify's plugin mechanism lets you hook your team's own capabilities into the build phase — for example content transformation, navigation enhancement, virtual module injection, search index generation, translation pipelines, or post-build artifact processing.

If you are only writing docs, you do not need to understand plugins; if you want to integrate Clarify into a product-level documentation platform or an internal engineering system, plugins are the recommended extension point.

---

## When to use plugins

Scenarios well suited to plugins:

| Scenario | What you can do |
|------|------------|
| Content governance | Uniformly inject disclaimers, validate heading structure, fill in metadata |
| Navigation enhancement | Reorder routes by team rules, filter internal pages, supplement navigation groups |
| Search integration | Generate a Pagefind, Algolia, or custom search index during the build |
| Internationalization workflow | Sync translation artifacts before and after the build, or check for missing translations |
| Site integration | Output a sitemap, RSS, analytics config, or an internal platform manifest |

Things plugins are not suited for:

- Layout of a single article's content: prefer MDX.
- Common site title, navigation, and theme color: prefer `clarify.ts` configuration.
- Runtime interactive components: prefer integrating them as React/MDX components.

---

## Basic structure

A plugin is an object with a `name` and `hooks`:

```ts
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const auditPlugin: ClarifyPlugin = {
  name: 'docs-audit',
  hooks: {
    'routes:resolved'({ routes, navigation }) {
      console.log(`Resolved ${routes.length} routes`)
      return { routes, navigation }
    },
  },
}

export default defineConfig({
  title: 'Product Docs',
  plugins: [auditPlugin],
})
```

> Use `clarify.ts` when you need plugins. `clarify.json` cannot express functions, so it is not suitable for configuring plugins.

---

## Lifecycle

Clarify's current build flow calls the following hooks:

| Hook | When it fires | Typical use |
|------|----------|----------|
| `routes:resolved` | After routes and navigation are generated | Adjust routes, navigation, groups, and visible pages |
| `modules:before` | After virtual modules are generated, before handing them to Vite | Inject or modify runtime virtual modules |
| `build:done` | After the build and SSG complete | Generate extra artifacts, report build results |

Plugins run in the order of the `plugins` array. The result returned by one plugin becomes the input to the next.

> The types also reserve `pages:resolved` and `page:transform`, but the current CLI build flow does not yet trigger them. Plugin authors should prefer the hooks in the table above.

---

## Hook context

Each hook receives a unified context object:

```ts
type ClarifyHookContext = {
  projectConfig: ResolvedProjectConfig
  generateOptions: ResolvedBuildOptions
}
```

- `projectConfig`: the site config after defaults have been applied, e.g. `title`, `routePrefix`, `i18n`.
- `generateOptions`: the build run parameters, e.g. the content directory, output directory, and SSG error strategy.

## Example: filter draft pages

You can filter out pages you do not want to publish during the `routes:resolved` stage:

```ts
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const hideDraftRoutes: ClarifyPlugin = {
  name: 'hide-draft-routes',
  hooks: {
    'routes:resolved'({ routes, navigation }, ctx) {
      if (ctx.generateOptions.ssg.failOnError === false) {
        return { routes, navigation }
      }

      return {
        routes: routes.filter(route => !route.path.includes('/drafts/')),
        navigation,
      }
    },
  },
}

export default defineConfig({
  plugins: [hideDraftRoutes],
})
```

If you modify both `routes` and `navigation`, you must keep them consistent to avoid the sidebar pointing to nonexistent pages.

## Example: generate artifacts after the build

`build:done` is well suited to generating extra files, such as a sitemap, search index, or internal platform manifest:

```ts
import { writeFileSync } from 'node:fs'
import { join } from 'node:path'
import { defineConfig, type ClarifyPlugin } from '@clarify-labs/cli'

const buildManifestPlugin: ClarifyPlugin = {
  name: 'build-manifest',
  hooks: {
    'build:done'(ctx) {
      const output = ctx.generateOptions.outputDirectory ?? 'output'
      writeFileSync(
        join(output, 'clarify-build.json'),
        JSON.stringify({ title: ctx.projectConfig.title }, null, 2),
      )
    },
  },
}

export default defineConfig({
  plugins: [buildManifestPlugin],
})
```

## Error handling

If a plugin hook throws, Clarify aborts the current flow and reports an error message that includes the plugin name and hook name:

```txt
[clarify] plugin "my-plugin" hook "routes:resolved" failed: Error: ...
```

Suggestions for plugin authors:

- Give the plugin a stable, readable `name`.
- Inside a hook, only do the work necessary for that stage.
- When modifying routes, navigation, or virtual modules, return the complete result.
- Set timeouts and fallback strategies for external service calls.
