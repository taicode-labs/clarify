# API Documentation

This page brings together Clarify's OpenAPI capabilities: generating a complete API reference, embedding a single endpoint in MDX guides, organizing product-level API docs, and outputting the raw spec for AI tools.

Clarify treats OpenAPI specs as a first-class content source. The spec is responsible for endpoint facts, while MDX explains the business workflows.

---

## What you can do with it

| Goal | Recommended capability |
|------|----------|
| Publish a complete endpoint list | `.openapi.json` / `.openapi.yaml` auto-generates the API reference |
| Show a single endpoint in a tutorial | `OpenApiOperation` |
| Add context before and after a full reference | `OpenApiDocument` |
| Split API pages by business domain | `filter.tags` generates multiple grouped pages from the same spec |
| Help developers quickly try a call | Automatically displays servers, server variables, auth inputs, request bodies, and response examples |
| Copy runnable request code | Generates multi-language, multi-client code examples from the spec |
| Let AI / LLMs read the raw spec | The `.openapi.*` raw files in the build output and `llms.txt` |
| Maintain API docs for multi-language sites | Place the spec in a language directory, e.g. `source/zh-CN/api.openapi.json` |

OpenAPI pages are more than static field tables. Clarify turns the spec into interactive, developer-oriented endpoint documentation: users can select a server, fill in server variables, choose an auth method, switch between request and response examples, and copy request code in different languages.

## Minimal example

Place an OpenAPI file in the content directory:

<FileTree aria-label="Minimal OpenAPI content layout">
  <FileTreeItem name="source" type="folder">
    <FileTreeItem name="index.mdx" />
    <FileTreeItem name="api.openapi.json" />
  </FileTreeItem>
</FileTree>

Clarify generates:

| Artifact | Description |
|------|------|
| `/api` | The complete API reference page |
| `/api.openapi.json` | The raw OpenAPI spec output |
| The OpenAPI link in `llms.txt` | Makes it easy for AI tools to discover and read |

If you maintain the sidebar explicitly, add this to `tabs`:

```ts
{
  openapi: 'api.openapi.json',
  title: 'API Reference',
  icon: 'FileJson2',
}
```

You can also publish multiple example pages from the same OpenAPI file via `filter.tags`. A filtered page only shows endpoints that match at least one of the specified tags in the OpenAPI `operation.tags`:

```ts
{
  openapi: 'api.openapi.json',
  path: 'api/projects',
  title: 'Example: Projects API',
  icon: 'FolderKanban',
  filter: {
    tags: ['Projects'],
  },
}
```

For example, after configuring `filter.tags: ['Projects']` for `api.openapi.json`, the generated page route defaults to `/api/projects`. Tags are converted to lowercase kebab-case; multiple tags are joined in configuration order, e.g. `['Admin Users', 'Billing']` generates `/api/admin-users-billing`. Set `path` when you want to choose the final route explicitly; the router uses `path` as the published URL.

## Spec formats and routing

OpenAPI 3.0 / 3.1 is supported, with these file extensions:

| Format | Example | Route |
|------|------|------|
| JSON | `api.openapi.json` | `/api` |
| YAML | `billing.openapi.yaml` | `/billing` |
| YAML shorthand | `admin.openapi.yml` | `/admin` |

When i18n is enabled, OpenAPI files also follow the language-directory rule. The default language has no locale prefix, while non-default languages include the locale prefix:

<FileTree aria-label="Localized OpenAPI files">
  <FileTreeItem name="source" type="folder">
    <FileTreeItem name="zh-CN" type="folder">
      <FileTreeItem name="api.openapi.json" description="-> /api" />
    </FileTreeItem>
    <FileTreeItem name="en-US" type="folder">
      <FileTreeItem name="api.openapi.json" description="-> /en-US/api" />
    </FileTreeItem>
  </FileTreeItem>
</FileTree>

So when embedding an endpoint in a Chinese page you typically use `specPath="/api"`; when embedding the same spec in an English page you typically use `specPath="/en-US/api"`.

At build time, Clarify uses `@apidevtools/swagger-parser` to parse and dereference the spec. When the spec is invalid, the production build fails, preventing the publication of stale or incorrect endpoint docs.

## Embedding endpoints in guides

Product-level API documentation should not be just a reference. We recommend writing guides around real tasks and embedding the corresponding operation next to the key steps.

```mdx
# Authentication

Use an access token to call protected endpoints. After the token expires, recreate the session.

<OpenApiOperation specPath="/api" path="/sessions" method="post" />
```

`OpenApiOperation` parameters:

| Parameter | Description |
|------|------|
| `specPath` | The OpenAPI route, without the `.openapi.json` suffix, e.g. `/api` |
| `path` | The path in the OpenAPI `paths`, which must match exactly |
| `method` | The HTTP method, lowercase is recommended, e.g. `get`, `post`, `patch` |

To insert a complete reference in MDX, use:

```mdx
<OpenApiDocument specPath="/api" />
```

## The interactive endpoint panel

`OpenApiOperation` and the full API reference automatically generate an endpoint panel from the spec. The more complete the spec, the more useful the on-page interactions:

| Capability | Source | What the user sees |
|------|------|--------------|
| Server selection | `servers` | Switch between different environments or domains |
| Server variables | `servers[].variables` | Fill in or choose variable values such as region or version |
| Auth inputs | `components.securitySchemes` + `security` | Placeholder inputs generated based on API Key, Bearer, Basic, OAuth, and other schemes |
| Parameter descriptions | `parameters` | Path, query, and header parameters shown in groups |
| Request body examples | `requestBody.content.*.example/examples/schema` | Switch between media types and examples |
| Response examples | `responses` | Show different status codes and response bodies |
| Copy request code | OpenAPI spec + current selection | Copy the request example for the current endpoint in one click |

Clarify generates request code for common languages and clients, including cURL, Wget, HTTPie, JavaScript/TypeScript Fetch, Axios, ofetch, Python Requests, HTTPX, aiohttp, Go, Java, C#, PHP, Ruby, Rust, Swift, Kotlin, and PowerShell.

  <FileTreeItem name="source" type="folder">
    <FileTreeItem name="guides" type="folder">
      <FileTreeItem name="authentication.mdx" />
      <FileTreeItem name="pagination.mdx" />
      <FileTreeItem name="webhooks.mdx" />
    </FileTreeItem>
    <FileTreeItem name="api.openapi.json" />
    <FileTreeItem name="index.mdx" />
  </FileTreeItem>
</FileTree>

Authoring responsibilities:

| Role | Responsible for |
|------|----------|
| Backend / API platform | The OpenAPI spec, schemas, response codes, and auth methods |
| Technical writing | Guide structure, call flows, and error explanations |
| Frontend / SDK | Example code, SDK behavior, and real integration scenarios |
| Product / DevEx | Information architecture, naming, and reader task paths |

Pre-publish checklist:

- The OpenAPI file parses successfully in the build.
- There is a clear API reference entry in the sidebar.
- Key business workflows have guides, not just an endpoint list.
- Example requests and responses contain no leaked test data.
- Error responses are clearly documented.
- `llms.txt` lists the raw OpenAPI spec link.

## Content quality suggestions

The quality of an API reference depends largely on the OpenAPI spec itself. We recommend at least filling in:

- `info.title`, `info.version`, `info.description`
- `servers`, and `servers[].variables` if there are variables
- `tags`, used for navigation grouping and `filter.tags` page splitting
- `summary` and `description` for each operation
- `description`, `schema`, `example`, or `examples` for parameters
- The media type, `example`, `examples`, or schema (from which examples can be derived) for request bodies and responses
- Major response codes and error responses
- Reusable `components.schemas`
- The auth methods `components.securitySchemes` and operation/security requirements

## FAQ

<AccordionGroup>
  <Collapse title="The API page does not appear in the sidebar?">
    If you explicitly configure `tabs[].pages`, Clarify does not automatically insert all files into the sidebar. You need to add the `openapi` entry manually.
  </Collapse>
  <Collapse title="OpenAPI spec not found?">
    Check whether the OpenAPI file is under `source`, whether its name uses `.openapi.json`, `.openapi.yaml`, or `.openapi.yml`, and whether `specPath` is written as a route rather than a file name.
  </Collapse>
  <Collapse title="Endpoint not found?">
    Check whether `path` matches the OpenAPI `paths` key exactly, whether path parameters use the `{id}` form, and whether `method` exists under that path.
  </Collapse>
</AccordionGroup>
